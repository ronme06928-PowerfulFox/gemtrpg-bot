1. シナリオ本文、敵データ、スキル・パッシブ・バフの実データ、テストプレイ、公開準備は本書で管理する。
2. シナリオ固有の名前や挙動をアプリへハードコードしない。
3. 汎用コードの追加・変更が必要なら、実装仕様と受け入れ条件を定めた別計画書を作る。
4. 別計画書には本書をユースケースとして記載し、本書の「連携計画」からも参照する。
5. 実装済みの汎用仕様は`manuals/implemented/`へ移管する。シナリオ実データはシナリオ本文またはスプレッドシートを正とする。
6. シナリオ本文、実データ、検証、公開準備が完了し、未完了の連携計画がなくなった時点で本書を削除する。

## 4. 決定済み仕様

### 4.1 シナリオ全体

- 想定プレイ時間は約3時間。
- 基本参加人数は3人、4人への拡張も考慮する。
- 実戦闘は通常戦闘2回とボス戦1回。その他の小競り合いは探索判定による擬似戦闘として処理する。
- 成長量は通過点1〜3、シナリオ経験は2前後を基準とする。
- ボス戦は約15ラウンドを基準とし、長くても20ラウンド程度を許容する。

### 4.2 研究者

| 記号 | 名前 |
|---|---|
| A | ファルア |
| B | ベター |
| C | タデル |
| D | マガン |

役割、負傷状態、救出順、脱出への寄与は未決定。

### 4.3 ボス戦の瓦礫ギミック

- ボス1体の横に、行動しないキャラクターとして瓦礫を配置する。
- 瓦礫は一定ラウンドごとに補充する。正確な補充間隔と上限数は未決定。
- PCが瓦礫を破壊した場合、次ラウンドのみPC陣営全員のスキル最終威力を+1する。データ上は`delay=1 / lasting=1`を使う。
- ボスが瓦礫を破壊した場合、次ラウンドから2ラウンド、ボスのスキル最終威力を+1する。データ上は`delay=1 / lasting=2`を使う。
- 破壊報酬の分岐は瓦礫側の死亡時パッシブに定義し、破壊したキャラクターと使用スキルを参照する。
- ボスのHPが一定値まで減ると広域・個別攻撃を使い、場の瓦礫も破壊する。発動条件と攻撃データは未決定。

### 4.4 スキル「鉄塊を捥ぎり喰う」

- 威力は`9+1d1`。
- 3ラウンドに1回、3R・6R・9R……で使用する。
- `target_scope`は`any`とする。
- 第一候補は同陣営かつ有効な`瓦礫`タグを持つキャラクター、第二候補は生存PCとする。
- 第一候補が空なら同じスキルのまま第二候補を狙う。
- 全候補が空の状態はPC全滅として通常の敗北処理へ委ねる。
- プレイヤーは通常どおり割り込み、マッチできる。
- 対象が`瓦礫`タグを持つ場合だけ`DEAL_TARGET_MAX_HP_DAMAGE`を適用する。対象がPCなら通常威力だけを適用し、最大HPダメージへフォールバックしない。

## 5. 制作タスク

### 5.1 シナリオ本文

- [ ] 導入、探索1、通常戦闘2回、探索2、ボス戦、脱出、エンディングの最終構成を確定する。
- [ ] 探索判定による擬似戦闘の成功・失敗結果を定義する。
- [ ] 研究者4人の役割、状態、情報、救出順を定義する。
- [ ] 脱出判定の難易度、支援条件、失敗時戦闘を確定する。
- [ ] 基本報酬と追加報酬の条件を確定する。

### 5.2 敵・戦闘データ

- [ ] 通常戦闘1・2の敵構成、HP、各種ステータス、スキルを確定する。
- [ ] ボスのHP、各種ステータス、通常技、強硬技、大技を確定する。
- [ ] 瓦礫キャラクターのHP、タグ、表示情報を確定する。
- [ ] 「鉄塊を捥ぎり喰う」の正式スキルJSONとIDを作成する。
- [ ] PC側・ボス側の威力バフと、瓦礫死亡時パッシブの正式ID・JSONを作成する。
- [ ] 3R周期と順序付き対象候補を持つボス行動フローチャートを作成する。
- [ ] HP低下時の広域・個別攻撃と瓦礫破壊処理を定義する。

### 5.3 実機確認

- [ ] 3R・6R・9Rで「鉄塊を捥ぎり喰う」が選ばれる。
- [ ] 瓦礫が存在する場合は瓦礫を、存在しない場合は生存PCを選ぶ。
- [ ] PCが割り込んだ場合、通常のマッチとして処理される。
- [ ] 瓦礫への命中時だけ最大HPダメージが発生する。
- [ ] PCへの命中時は通常の`9+1d1`だけが発生する。
- [ ] HP0の瓦礫へ追加ダメージが発生しても死亡時パッシブが再発しない。
- [ ] PC破壊時とボス破壊時で正しいバフ、対象、開始ラウンド、持続回数になる。
- [ ] 敵インテント矢印、redirect、マッチ対象表示が正しい。

### 5.4 テストプレイと公開

- [ ] 3人構成で通しテストを行い、所要時間、各戦闘ラウンド、消耗量、救出結果を記録する。
- [ ] 4人構成の敵HP・構成・行動数調整を決める。
- [ ] 初心者へ過度なラウンド圧力を与えない進行になっているか確認する。
- [ ] GM向けデータ作成・導入手順を整理する。
- [ ] 配布用シナリオ本文と必要データをまとめる。
- [ ] Discordでの掲載場所、更新告知、版管理方法を決める。
