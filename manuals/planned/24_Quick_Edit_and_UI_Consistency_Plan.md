# 24. 簡易ステータス編集改善 / UI統一計画
作成日: 2026-05-07

## 1. 目的
本計画は次の2件を分離して扱う。

1. 簡易ステータス編集機能の操作性改善
2. アプリケーション全体のUI統一

理由は、前者は既存機能の実用性を上げる局所改善であり、後者は画面横断のルール整備と段階的な置換を伴うため、スコープとレビュー基準を分ける必要があるため。

---

## 2. 対象範囲

### 2.1 先行実装対象
- `static/js/action_dock.js`
- 必要に応じてバフ/アイテム辞書参照部
- 簡易ステータス編集モーダル内のGM操作UI

### 2.2 別件扱い対象
- Visual Battle 全体のカード、フォーム、ボタン、見出し、余白
- 既存モーダル群の共通デザイン
- CSSモジュールの命名と責務整理
- コンポーネント単位の再利用UI整備

## 3. ストリームA: 簡易ステータス編集改善 ✅ 実装完了（2026-05-08 コード確認済み）

## A-1. 目的
GMが `buff_id` を暗記していなくても、辞書から名称と効果を確認しながら付与/解除できる状態にする。加えて、`lasting` `delay` `count` の意味をUI上で明示し、誤操作を減らす。

## A-2. 要件
- バフ付与時に `buff_id` 手入力を基本操作にしない
- バフ選択UIから `buff_id / 名称 / 説明` を確認できる
- Value駆動バフに `data.value` を与えられる
- 継続ラウンド、ディレイ、スタック数の意味がUI上で読める
- 解除時に現在付与中のバフを識別しやすい
- 出血 / 破裂 / 亀裂 / 戦慄 / 荊棘 をこの画面から付与できる
- 亀裂はスタック数に加えて継続ラウンド数を設定できる
- 既存のSocket API仕様は極力維持し、必要な追加はGM専用イベントに限定する

## A-3. 実装段階

### Phase A1: 辞書連動の選択UI
- バフ付与欄をプルダウン化
- `window.buffCatalogData` を参照して候補生成
- 候補表示形式を `buff_id + 名称` に統一

### Phase A2: 選択内容プレビュー
- 選択中バフの名称、説明、効果要約を表示
- `default_duration` がある場合は既定継続値も表示
- `lasting / delay / count` の現在設定をプレビューに反映

### Phase A3: 解除UI改善
- 付与中バフ一覧を `名称 [buff_id] (継続/待機/スタック)` 形式で表示
- 選択中の解除対象もプレビュー表示
- `buff_id` 不明なエントリは解除不能として明示

### Phase A4: 入力補助
- `lasting / delay / count` のラベル整理
- Value駆動バフ向けの `value` 入力欄を追加
- 状態異常付与欄を追加
- アイテム側も辞書プルダウン化して説明と所持数を確認できるようにする

## A-4. 受け入れ条件
- GMが `Bu-xx` を覚えていなくてもバフ付与できる
- Value駆動バフに `data.value` を指定できる
- バフ解除時に対象の識別で迷いにくい
- 継続/待機/スタックの意味が画面上で説明される
- 状態異常をこの画面から直接付与できる
- 亀裂の継続R付き付与をこの画面から実行できる
- 既存の `request_gm_apply_buff` / `request_gm_remove_buff` / `request_gm_adjust_item` を維持する
- 追加イベントは GM専用の `request_gm_apply_state` に限定する

## A-5. リスク
- 辞書説明の長文でモーダル密度が上がる
- 一部バフは `effect` が構造化されていても人間向け要約が弱い
- キャラ実体に `buff_id` が無い古いデータは解除UIで扱いづらい
- Value駆動バフのうち `data.value` 必須のものは、値未指定時に実行時エラーになりうる
- 状態異常付与と通常ステータス編集の責務境界が曖昧になると、将来のUI整理で再分離が必要になる

## A-6. 現在の実装到達点
- バフ付与欄は辞書連動プルダウン化済み
- 選択中バフの説明・要約プレビューは実装済み
- 解除欄の識別性改善とプレビューは実装済み
- Value入力欄は実装済み
- 出血 / 破裂 / 亀裂 / 戦慄 / 荊棘 の付与UIは実装済み
- 亀裂の継続R付き付与は `request_gm_apply_state` 経由で実装済み
- アイテム増減欄の辞書プルダウン化は実装済み
- GM Buff / Item Control 欄の折り畳み化は実装済み
- 内側の重複見出し削除は実装済み
- コード上の最低限チェックは完了
  - `node --check static/js/action_dock.js`
  - `python -m py_compile events/socket_char.py`
- 実ブラウザ確認は今回は完了条件から外し、必要時に任意確認とする
- ストリームAは `実装完了・軽検証完了` として扱い、以後は不具合修正のみ行う

## 4. ストリームB: アプリ全体のUI統一

## B-1. 目的
戦闘系画面を中心に「同じ役割のUIは同じ見た目と挙動にする」状態へ寄せ、TRPGツールとしての統一感を出す。

## B-2. デザイン原則（決定済み）

| 軸 | 決定内容 |
|---|---|
| 第一目的 | 見た目の統一感（次点: 誤操作削減） |
| 対象スコープ | 戦闘系画面のみ（Action Dock・戦闘モーダル群） |
| 基調トーン | 少し演出寄り（TRPGらしさを業務操作を妨げない範囲で出す） |
| 情報密度 | 折り畳みで可変（デフォルトは広め、展開で詳細） |
| ベースカラー | ダーク地（#222）＋明モーダル（#fff）の2ゾーン維持 |

## B-3. デザイントークン（決定済み）

```css
/* ── 背景 ── */
--bg-dark:       #222222;   /* 戦闘マップ・Action Dock */
--bg-dark-mid:   #2e2e3a;   /* ダーク面のカード・パネル */
--bg-light:      #f8f9fa;   /* サイドバー・詳細パネル */
--bg-modal:      #ffffff;   /* モーダル */

/* ── アクセント ── */
--accent-gold:   #c8a84b;   /* 物理・攻撃・主要操作ボタン・見出しライン */
--accent-cyan:   #26c6da;   /* 魔法・情報・プレビュー枠・展開トリガー */

/* ── 機能色 ── */
--color-hp:      #28a745;
--color-mp:      #4a90d9;
--color-fp:      #e0b84a;
--color-danger:  #dc3545;
--color-neutral: #6c757d;

/* ── テキスト ── */
--text-on-dark:  #e8e8e8;
--text-on-light: #333333;
```

## B-4. コンポーネント規約（決定済み）

| 項目 | 規則 |
|---|---|
| ボタン種別 | primary(Gold) / secondary(Cyan) / ghost / danger の4種。disabled は opacity で表現 |
| 角丸 | ボタン 6px / カード・モーダル 8px / バッジ 12px |
| モーダルヘッダー | 白背景＋上辺に `--accent-gold` の 4px ボーダーライン |
| 反映タイミング | 表示切替・折り畳み=自動反映 / キャラデータへの書き込み=明示送信 |
| 破壊的操作の確認 | バフ解除・状態異常削除=確認なし / HP・MPを0にする=確認あり / キャラ削除・戦闘終了=確認あり（固定） |

## B-6. PR分割方針
- 1PRの変更範囲: 見た目変更＋インライン style 外出し・クラス名整理まで許容
- 機能変更（ロジック・イベント）は別PRに分離
- 対象ファイルが3つ以上になる場合はPRを分割する

## B-7. 非目標
- トップページ・非戦闘画面への適用（別フェーズ）
- JSフレームワーク導入・大規模コンポーネント再設計
- 既存仕様変更を目的とした変更

## B-8. 受け入れ条件
- 同じ種類の入力部品が同じ見た目で振る舞う
- モーダル間で見出し・セクション・ボタンの階層感が揃う
- インライン style の比率が下がり CSS モジュール側で再利用可能になる
- デザイントークンが `:root` に集約されており、色変更が1箇所の編集で全体に反映される
