# 24. 簡易ステータス編集改善 / UI統一計画
作成日: 2026-05-07

## 1. 目的
本計画は次の2件を分離して扱う。

1. 簡易ステータス編集機能の操作性改善
2. アプリケーション全体のUI統一

理由は、前者は既存機能の実用性を上げる局所改善であり、後者は画面横断のルール整備と段階的な置換を伴うため、スコープとレビュー基準を分ける必要があるため。

---

## 2. 対象範囲

### 2.1 先行実装対象
- `static/js/action_dock.js`
- 必要に応じてバフ/アイテム辞書参照部
- 簡易ステータス編集モーダル内のGM操作UI

### 2.2 別件扱い対象
- Visual Battle 全体のカード、フォーム、ボタン、見出し、余白
- 既存モーダル群の共通デザイン
- CSSモジュールの命名と責務整理
- コンポーネント単位の再利用UI整備

---

## 3. ストリームA: 簡易ステータス編集改善

## A-1. 目的
GMが `buff_id` を暗記していなくても、辞書から名称と効果を確認しながら付与/解除できる状態にする。加えて、`lasting` `delay` `count` の意味をUI上で明示し、誤操作を減らす。

## A-2. 要件
- バフ付与時に `buff_id` 手入力を基本操作にしない
- バフ選択UIから `buff_id / 名称 / 説明` を確認できる
- 継続ラウンド、ディレイ、スタック数の意味がUI上で読める
- 解除時に現在付与中のバフを識別しやすい
- 既存のSocket API仕様は極力変更しない

## A-3. 実装段階

### Phase A1: 辞書連動の選択UI
- バフ付与欄をプルダウン化
- `window.buffCatalogData` を参照して候補生成
- 候補表示形式を `buff_id + 名称` に統一

### Phase A2: 選択内容プレビュー
- 選択中バフの名称、説明、効果要約を表示
- `default_duration` がある場合は既定継続値も表示
- `lasting / delay / count` の現在設定をプレビューに反映

### Phase A3: 解除UI改善
- 付与中バフ一覧を `名称 [buff_id] (継続/待機/スタック)` 形式で表示
- 選択中の解除対象もプレビュー表示
- `buff_id` 不明なエントリは解除不能として明示

### Phase A4: 入力補助
- `lasting / delay / count` のラベル整理
- 必要なら speed系など頻出バフ用の簡易テンプレートを追加
- 将来的に item 側も辞書プルダウン化するかを再判定

## A-4. 受け入れ条件
- GMが `Bu-xx` を覚えていなくてもバフ付与できる
- バフ解除時に対象の識別で迷いにくい
- 継続/待機/スタックの意味が画面上で説明される
- 既存の `request_gm_apply_buff` / `request_gm_remove_buff` / `request_gm_adjust_item` を維持する

## A-5. リスク
- 辞書説明の長文でモーダル密度が上がる
- 一部バフは `effect` が構造化されていても人間向け要約が弱い
- キャラ実体に `buff_id` が無い古いデータは解除UIで扱いづらい

## 4. ストリームB: アプリ全体のUI統一

## B-1. 目的
画面ごとに異なる見た目と操作感を整理し、Visual Battle 系UIを中心に「同じ役割のUIは同じ見た目と挙動にする」状態へ寄せる。

## B-2. 進め方
UI統一は一括刷新ではなく、ルール定義 -> 共通部品化 -> 対象画面の順で進める。

## B-3. 実装段階

### Phase B0: 現況棚卸し
- 対象画面の洗い出し
  - Action Dock
  - 各種 modal
  - 戦闘パネル
  - キャラクター詳細表示
- 不統一項目の分類
  - 色
  - ボーダー半径
  - ボタン形状
  - 見出し階層
  - 余白
  - フォーム部品
  - バッジ/状態表示

### Phase B1: UIルール定義
- 色の役割を固定
  - primary
  - success
  - warning
  - danger
  - neutral
- フォーム寸法、角丸、境界線、フォーカス時の見え方を固定
- カード、モーダル、セクション見出しの規則を定義
- HP/MP/FP とバフ系表示の色ルールを文書化

### Phase B2: 共通CSS整理
- `static/css/modules/common_ui.css` に寄せる責務を整理
- `action_dock.js` 内のインライン style を段階的に外出し
- 再利用する class 命名規則を定める

### Phase B3: コンポーネント単位の移行
- まず簡易ステータス編集UI
- 次に他モーダル
- 最後に戦闘中の高頻度操作UI

### Phase B4: 実機確認
- PC幅
- 狭い横幅
- 長い名称/長文説明
- バフ件数が多いケース

## B-4. 非目標
- この計画ではトップページや非戦闘画面まで一気に揃えない
- JSフレームワーク導入や大規模なコンポーネント再設計までは行わない
- 既存仕様を変えること自体を目的にしない

## B-5. 受け入れ条件
- 同じ種類の入力部品が同じ見た目で振る舞う
- モーダル間で見出し、セクション、ボタンの階層感が揃う
- インライン style の比率が下がり、CSSモジュール側で再利用可能になる

## 6. 直近の実務順

1. ストリームAを完了させる
2. UI統一用のルール文書を先に確定する
3. `action_dock.js` のインライン style を外へ出す
