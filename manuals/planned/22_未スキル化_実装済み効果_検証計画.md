# 22. 未スキル化の実装済み効果 検証計画

**作成日**: 2026-05-02  
**目的**: 実装済みだが運用検証が不足している「スキルカテゴリ封印 / コスト増加デバフ / フィールド効果制約」を、検証用スキルで重点テストする。

---

## 1. 対象選定基準
1. サーバ実装に処理分岐が存在する。  
2. `skills_cache` の `effects[].type` / `CUSTOM_EFFECT` 値に未出現。  
3. 将来的にスキル化または内部仕様確定の価値がある。

---

## 2. 検証対象（重点）
### 2.1 スキルカテゴリ封印（block）
1. `category=物理` の封印  
2. `category=魔法` の封印  
3. `category=補助` の封印  
4. 複数カテゴリ混在時の評価順・競合解決

### 2.2 コスト増加デバフ（add_cost）
1. `cost_types=["FP"]` への加算  
2. `cost_types=["MP"]` への加算  
3. 複数制約同時適用時の合算  
4. コスト不足時の使用不可判定と理由表示

### 2.3 フィールド効果由来の制約
1. `state.field_effects[].skill_constraints` の適用  
2. `stage_field_effect_profile.rules[].skill_constraints` の適用  
3. 個別デバフ由来制約との優先順位  
4. 同一 `id` 競合時エラー（重複検知）

---

## 3. 期待仕様を先に固定すべき項目
1. `match.category` の判定元  
- スキル本体 `分類` と `rule_data` のどちらを優先するか。

2. `add_cost` の適用順  
- 複数ルールで加算する順番と合算方式（単純和）を明文化。

3. 制約衝突時の優先順位  
- `block` と `add_cost` が同時命中する場合の扱い。  
- フィールド制約と個別デバフ制約の競合時優先順位。

---

## 4. 検証ステップ
1. `block` / `add_cost` / `field_constraints` の3系統で最小スキルJSONを作成。  
2. 敵専用テストスキルとして1条件ずつ実機検証。  
3. 失敗時は `evaluate_skill_access` の判定ログと照合。  
4. 安定化後、本番用デバフスキルへ順次展開。

---

## 5. 生成物
1. カテゴリ封印用 検証JSON（2行版）  
2. コスト増加用 検証JSON（2行版）  
3. フィールド制約用 検証JSON（2行版）  
4. 実測ログ（成功/失敗/差分）

---

## 6. 完了条件
1. 2章の重点対象に対して全ケース結果がある。  
2. 3章の曖昧点が解消され、仕様文書へ反映済み。  
3. 本番投入可能なテストスキル定義が確定している。
