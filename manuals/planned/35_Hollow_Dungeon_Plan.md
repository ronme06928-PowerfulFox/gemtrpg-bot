# 35 ホロウダンジョン計画（ハクスラ×不思議のダンジョン）

**作成日**: 2026-07-10
**最終更新**: 2026-07-11（残る§10論点をほぼ全て推奨方針で決定。特に「ホロウ内で完結の原則」（アイテムの持ち出しは一切なし。持ちキャラへの反映は経験値のみ）を決定し、報酬テーブル・撤退時の扱いを修正。未決定は準備作業・技術検証のみ3件）
**位置づけ**: 世界設定『アヌッサ・ホロウ』を題材にした階層制ハクスラコンテンツの追加計画。
アイテム・淵遺物・バフを獲得しながら下の階へ進み、深くなるほど強化された敵と戦う
「遠征（ラン）」型のプレイモードを、既存の battle_only（PvE）基盤の上に構築する。
本計画の対象は**浅層 1F〜50F のみ**。51F以降の深層は「危険につき到達不能」のフレーバーで封鎖し、実装しない。
**依存**: 味方キャラの持ち込みは計画36（キャラクター管理基盤）の持ちキャラ投入を正式経路とする。

---

## 1. 目的

- GMの重い準備なしで繰り返し遊べる高再現性の戦闘コンテンツ（ハクスラ）を提供する。
- 「階を降りるごとに敵が強くなる／報酬で自分も強くなる」の成長ループを、既存の戦闘エンジンとバフ/アイテム基盤の再利用で実現する。
- ストーリークリア後のエンドコンテンツとして、既存キャラビルドの持ち込み先を作る。
- 深層（51F〜）・神話レギュレーション・魔女化などの将来拡張に耐えるデータ構造を先に敷く。
- 通常のTRPGルーム運営とは独立した、ロビーから直接入れる専用のゲームモードとして提供する（詳細は§4.0）。

## 4. ゲームデザイン仕様（たたき台）

### 4.0 スペースの位置づけ（ロビー起点・専用ルーム・統合画面）

ホロウは**通常のTRPGルームの一機能ではなく、独立したモード**として提供する。

- **入口**: ロビー画面（`renderRoomPortal`が描く、通常ルーム一覧の画面）の**左下あたりに常設のホロウ入口ボタン**を追加する。通常ルームのカード一覧とは別の導線。
- **裏側の実装はRoom基盤を流用**: ゼロから認証・在室管理・membershipを作り直すコストを避けるため、内部的には既存の`Room`/`RoomMember`（join code・オーナー/GM・入退室）をそのまま使う。`Room.play_mode`に新しい値`'hollow'`を追加し、この種別のルームは**通常ロビーの一覧には出さない**（`build_lobby_cards`のフィルタ条件に追加）。プレイヤーからは「TRPGルームを作る/選ぶ」という操作を経由せず、ホロウ入口ボタンから直接この専用ルームの作成・参加（既存の参加コード機構を流用）に入る。
- **ダンジョンモード画面**: `play_mode == 'hollow'`のルームに入ると、通常ルームの画面（ドック＋通常の戦闘UI）ではなく**専用レイアウトの新規ビュー**を表示する。階層ハブ（現在階・イベント・パーティ）と**戦闘発生時のバトルUIを同一画面内に統合表示**する（別画面への遷移はしない）。
- **既存の戦闘コンポーネントは埋め込みで再利用**: 戦闘の見た目自体は`static/js/battle/`のESモジュール群（`ActionDock`/`DeclarePanel`/`MatchPanel`/`Timeline`/`VisualMap`等）をダンジョン画面のレイアウトへ**部品として埋め込む**形にし、ゼロから戦闘UIを作り直さない。バックエンドの戦闘解決ロジック（battle_only基盤）も変更なくそのまま使う。変わるのは「ページ全体を専用の戦闘画面に切り替えるか、ハブ画面の一部として表示するか」という統合の仕方だけ。
- **ホスト/パーティ/戦績などの既存設計への影響**: 「ルーム」であること自体は変わらないため、§4.1のホスト決定（`RoomMember`ベースの継承）や§4.7のパーティ編成（`party.order`）の設計はそのまま成立する。異なるのは「このルームは通常のTRPGセッション用ルームではなく、ホロウ専用に自動生成されたルームである」という点のみ。

### 4.1 ランの流れ

```
ラン開始（誰か1人が「ホスト」に立候補してパーティ確定・1Fから）
  └→ 階イベント抽選 → [戦闘系] battle_only PvE戦闘 → 勝利 → ドロップ抽選
                       [非戦闘系] 宝箱/魔素だまり/キャンプ を解決
      → 他プレイヤーは選択肢に「推薦ピン」を立てられる（強制力なし）
      → ホストが「次の階へ降りる」or「撤退する」を確定
      → 50Fボス撃破でクリア / 全滅で1Fから再挑戦 / 任意撤退
ラン終了（淵遺物を全て失う。持ち帰り対象の報酬を確定）
```

- ランの単位はルーム（§4.0の通り、通常のTRPGルームとは別に自動生成される`play_mode='hollow'`のルーム）。1ルームにつき同時に1ランのみ（`battle_id = f"battle_{room}"` 前提と整合）。
- **進行の主導権はプレイヤー側に置く**。GMなし・少人数（プレイヤーだけ）でも遊べることを前提にし、**ホロウ用のホスト**という役割を新設する（ラン単位ではなく**ルーム単位**で持続する設定。ランをまたいで引き継がれる）。ホストは既存のGM専用操作相当（battle_onlyの戦闘生成・編成等）をホロウに限って代行できる。
- ホスト以外のプレイヤーは、階イベントの選択肢（降りる/撤退、宝箱を開ける/開けない等）に**推薦ピン**を立てられる。ピンは非拘束（最終決定はホストが行う）で、他プレイヤーの意思をホストへ可視化するための軽量な合議UI。**将来的には淵遺物の選択（誰が取るか等）のような戦利品分岐にも推薦ピンを広げる方向**だが、対象拡大の詳細はゲーム仕様がより固まった段階（Phase 3〜4着手時）で改めて決める。初期実装（Phase 5）は降りる/撤退を中心としたイベント分岐に絞る。
- ラン状態はルーム状態に保存されるため、途中でセッションを終えても（全滅していない限り）次回同じ階から再開できる。

**ホストの決定方法**

- **既定値はルームオーナー**（`RoomMember.role == 'owner'` / `Room.owner_id`）。
- **明示的な譲渡**: 現ホストまたはGM/オーナーは、ダンジョンモード画面内のホスト設定から任意の在室メンバーへホストを譲渡できる（`request_hollow_set_host`）。これはオーナー権そのものの移動（既存の`transfer_owner`）とは独立しており、オーナーのままホストだけ他人に渡せる。
- **自動継承**: 現ホストがルームから退室した場合（`RoomMember`の`revoked_at`が打刻される、いわゆる明示的な退室/追放）、`RoomMember.joined_at`昇順で次の在室メンバーへホストを自動的に引き継ぐ。**単なるSocket切断（ブラウザを閉じただけ）では継承しない**——`handle_disconnect`はin-memoryの在室表示のみを更新し`RoomMember`には影響しないため、既存の挙動に合わせて「退室」と「一時切断」を区別する。
- **フォールバック（ホスト不在時）**: ホスト用の操作権限チェックは「`user_id == hollow.host_user_id` **または** GM/オーナーであること」の**OR条件**にする。これにより、ホストが未選任・離席中でもGMがそのまま代行できる。専用の待機/一時停止状態は用意しない。

### 4.2 階層構成（浅層 1F〜50F）

設定準拠: 30Fまでは各階にベースキャンプ、10/20/30Fは町、40/45/50Fは先人のキャンプ。

| 帯域（Tier） | 階 | 難度の目安 | 固定要素 | ビジュアルテーマ |
|---|---|---|---|---|
| T1 入穴 | 1〜9F | 地上の危険地帯並み（現行敵プリセットの標準帯） | — | 荒野・洞窟。まだ地上の延長線上で、異常性はまだ薄い |
| — | 10F | ボス | 撃破後: 町（完全回復。セッション中断→再開時の目印になる） | — |
| T2 浅層上部 | 11〜19F | T1の1.3〜1.5倍目安 | — | 異常な宝石などが見え始める。魔力濃度上昇の最初の兆候 |
| — | 20F | ボス | 撃破後: 町 | — |
| T3 浅層中部 | 21〜29F | 精鋭出現率上昇。精鋭セットがT2帯から切り替わる | — | マグマ地帯と寒冷地帯が隣接する異常環境。環境そのものが牙を剥き始める |
| — | 30F | ボス | 撃破後: 町 | — |
| T4 浅層下部 | 31〜39F | 魔素だまり（環境デバフ階）の出現率上昇 | 40F: ボス＋先人のキャンプ | 鉱脈と樹木が混ざった「森」。地下にあるはずのない生態系という違和感を演出する（決定 → §9。深く潜ってから森が現れることで、洞窟の延長という納得感を裏切り異常性を際立たせる狙い） |
| T5 深層境界 | 41〜49F | ステージ効果常設階が混ざる | 45F: 先人のキャンプ | 魔力生物化が主体に。獣の面影が失われていく |
| — | 50F | 浅層最深部ボス | 撃破でクリア。51Fへの道は崩落・封鎖のフレーバー表示 | — |

**生態系のグラデーション**（決定 → §9）: 敵の`タグ`列に、Tierが進むにつれて出現比率が変わる系統タグを用意する。`beast`（まだ獣）→`tainted`（変異が進んだ獣）→`arcane_beast`（魔力混じりの獣）→`aberration`（もはや獣の姿を保っていない異形）の4段階を想定し、浅い階層は「ちょっと異なる生態系のシュラーカーを狩る」感覚、深い階層は「濃すぎる魔力そのものと戦う異形との戦い」という感覚へ徐々に主軸を移す。

- 各階は開始時に**イベント種別を重み付き抽選**する（固定階を除く）。重みはTierごとにマスターデータで定義。
  - 通常戦闘（最頻）/ 精鋭戦闘（報酬良・敵強化）/ 宝箱（無戦闘ドロップ）/ **魔素だまり**（環境の澱み。§4.6参照。旧「残留思念」を吸収統合済み → §9）/ ベースキャンプ（部分回復 or 一時バフの二択。1〜30Fで低確率）。
- 10の倍数階のボスは階固定のマスターデータ（プリセットID＋行動チャート＋ステージ効果）で指定する。

### 4.3 敵編成とスケーリング

- 敵は**ホロウ専用の敵マスター**（`hollow_enemies`、§5.2）で管理する。battle_onlyの汎用プリセットとは独立させ、Tier・役割（雑魚/中堅/精鋭/ボス）・**等級**（鉄/銅/銀/金/特異級。既存設定のシュラーカー金属等級をそのまま流用 → §9）・出現重み・ドロップ補正といったホロウ固有の属性を敵定義自身に持たせる。
- **等級とTierの対応**（決定 → §9）: T1=鉄級／T2=銅級／T3=銀級／T4=金級／T5=特異級を基本対応としつつ、厳密な1:1固定ではなく**緩い相関**とする。各Tierの雑魚/中堅は基本的にそのTierの等級だが、**精鋭は基本的に1つ上の等級**（例: T1精鋭＝銅級個体）とし、さらに**低い階でも稀に上位等級の個体が出現しうる**（具体的な出現率・実装方式はPhase 2の`floor_builder.py`実装時に検討。データ上は各敵レコードに出現階範囲を持たせて対応する想定）。
- 敵の`名前`は既存のシュラーカー命名規則「**[地形]の[等級]級シュラーカー**」（例:「遺跡の銅級シュラーカー」「森の鉄級シュラーカー」）にホロウ内の地形（荒野/洞窟/異常な宝石地帯/マグマ地帯/寒冷地帯/鉱脈の森 等、§4.2のTierビジュアルテーマに準拠）を当てはめて統一する（決定 → §9）。
- **個体デザインの大まかな方向性**（決定 → §9）: シュラーカーは「鉱石核＋魔力吸収で成長する魔法生物」という既存設定に沿い、動物的なシルエット（四足・翼・尾等）は保ちつつ、核となった鉱物の質感（砂岩・鉱脈の欠片・鎖状結晶等）が体表に覗く程度に留める。等級が上がる（＝Tierが進む）につれて、この鉱物的な逸脱がより顕著になり、T5（特異級）帯では動物としての輪郭そのものが失われていく（§4.2の生態系グラデーションと対応）。具体的な個体名・詳細デザインはデータ作成時に別途検討する。
- 敵マスターの各レコードはプリセット互換の `character_json.data`（ステータス・`commands`・`behavior_profile`）を内包し、実体化は既存の `_build_runtime_character_from_preset` 相当の変換で流用する（実体化コードの二重実装を避ける）。
- 各階の編成は「Tierごとの編成テンプレート（例: 雑魚2＋中堅1）」から抽選し、Tier×役割×重みで敵マスターから個体を選んで `enemy_entries` 形式に変換し、既存の `handle_bo_start_battle` 相当の内部処理へ渡す。
- スケーリングは `_build_runtime_character_from_preset` で実体化した後に適用する:
  - HP/MP: `×(1 + k_hp × (floor - 1))`
  - 攻撃系パラメータ・スキル基礎値の補正: 直接の数値改変ではなく、**敵専用の階層バフ**（`stat_mod` 等の既存バフプラグイン）を `special_buffs` に積む方式を第一候補とする（スキルJSONを書き換えないため安全）。
  - 係数 `k_*` はマスターデータ側に置き、計画30のシミュレータで検証して調整する。
- **精鋭（エリート）はバフ積みを基本としつつ、階層が進むと「精鋭セット」自体が別データに切り替わる**方式にする。
  - `hollow_enemies` に「精鋭セット」単位（複数の敵IDをまとめた組、例: 「牙狼2体+大牙狼1体」）を定義し、各セットに**適用階範囲**（例: 21〜29F）を持たせる。精鋭イベントが抽選された階では、その階を含む範囲を持つ精鋭セットの中から抽選する。
  - 数階層ごと（Tier内のサブ帯域単位を想定。具体的な間隔は§10）にセットが切り替わることで、「同じ精鋭にバフが乗って強くなる」だけでなく「見た目・構成が変わる」体験を作る。
  - セット内の個体にも通常のスケーリングバフ（HP係数等）は重ねて適用する。

### 4.4 報酬（ドロップ・淵遺物・バフ）

**ホロウ内で完結する原則**: ホロウと通常のTRPGは根本的に別モードであるため、**アイテムの持ち出しはあらゆる面で一切行わない**（決定済み → §9）。アイテムドロップはラン中に使う戦術的資源であり、ラン終了（クリア/撤退/全滅のいずれでも）で**淵遺物と同様に消滅**する。持ちキャラ（計画36）へ反映されるのは経験値（成長）のみで、アイテムは反映対象外とする。

報酬は3系統に分ける。いずれもラン外への持ち出しはない。

| 系統 | 内容 | 実装 | ラン終了後 |
|---|---|---|---|
| 消耗品/アイテム | 既存アイテムマスターのアイテム。ラン中の戦術資源（回復薬等） | `grant_item` で `inventory` へ | **消滅**（ラン専用。持ち出し不可） |
| 淵遺物 | ラン中のみ有効なパッシブ強化。例:「与ダメ+10%」「毎ラウンドHP2回復」「出血無効」 | 新マスター `hollow_relics` ＋ 既存バフプラグイン（`stat_mod` 等）を `special_buffs` に恒常付与 | **消滅**（設定準拠:「地上では凡庸なアイテムと化す」） |
| 一時バフ | 数階層のあいだだけ有効な強化（キャンプの休息・残留思念の加護など） | 既存バフ＋残り階数カウンタ | 消滅 |
| — 参考: 通貨/素材の買取 | ルナ・バスティオン買取（経済） | 今回は扱わない（§2） | — |

- ドロップテーブルはレアリティ（コモン/レア/エピック/レジェンド）×Tierの重み表として定義。精鋭・ボス・宝箱はテーブルを別に持つ。
- 抽選結果はログに演出込みで出力し、淵遺物の付与は自動で行う（ホスト承認は既定でオフ、設定でオンにできる形を検討）。

**淵遺物の効果範囲（既定はパーティ全体）**

- 淵遺物は**既定でパーティ全体に効果を及ぼす**（`target_scope: "party"`）。個別キャラへの `special_buffs` 付与ではなく、パーティ共有のラン効果として管理し、戦闘実体化時にパーティ全メンバーへ同一のバフを適用する。
- 対象を絞る淵遺物は、淵遺物側の定義に**適用条件**か**編成順（スロット）**のいずれかを持たせる。
  - `target_scope: "conditional:<condition_id>"` — 例:「HPが最も低いキャラ」「前ラウンドで出血状態だったキャラ」など、既存の条件評価（`enemy_behavior.py` の条件式相当）を再利用。
  - `target_scope: "slot:N"` — ラン開始時に確定する**編成順**（`hollow.run.party_order` の並び）のN番目のキャラを対象にする（例: 先頭=1番目、2番目）。
  - スロット対象の淵遺物は、対象キャラがパーティにいない場合（脱落・不在）は効果なしとして扱う。
- パーティ全体対象の淵遺物はキャラ選択UIが不要なため、取得時に確定演出のみでよい。条件/スロット対象の淵遺物は取得時に「誰に効くか」をログで明示する（選択操作は不要）。

### 4.5 敗北・撤退・再挑戦

- **全滅**: ラン終了。淵遺物・アイテムは全て失う（§4.4の原則どおりラン外への持ち出しはない）。**再挑戦は常に1Fから**（チェックポイント再開はしない）。到達階数を戦績として記録する。
- **撤退**: 非戦闘時のみ選択可。全滅と同様、淵遺物・アイテムは全て失う（持ち帰れる報酬はない）。撤退後の再挑戦も1Fから。撤退の意味は「安全にランを終了できる」ことのみで、報酬の確定ではない。
- **中断からの再開（全滅ではない場合）**: ラン状態はルーム状態に保存され続けるため、セッションをまたいで**同じ階から**再開できる（10/20/30Fの町はこの再開時の目印として機能するのみで、リセット先ではない）。
- 戦績（到達階・クリア回数・撃破ボス）はルーム状態の `hollow.records` に蓄積する。

### 4.6 環境ギミック（魔素だまり・ステージ効果）

**魔素だまりは「残留思念」を吸収統合したイベント種別**とする（決定 → §9）。統合の理由は、両者とも「戦闘以外の寄り道イベント」という役割が重複しており、抽選重みやイベント解決コードを別々に持つ必要が薄いため。フレーバー（環境の澱みそのものか、澱みに宿った思念との対話/尋問か）は`ホロウ_イベント`シートの`定型文`列で描き分け、解決の仕組みは共通化する。

**解決フロー（共通）**: 魔素だまりに遭遇すると、**最大4つ（A〜D）まで選択肢を提示できる**（決定 → §9。実際に使う選択肢数はイベントごとに異なってよく、未使用の選択肢列は空欄のままにする。多くの場合は「関わる/やり過ごす」の二択で足りる想定だが、対話系の思念で「助ける/問い詰める/見過ごす」のような3択にする等、行ごとに柔軟に増やせる）。

各選択肢は独立して次のどちらかとして機能する:

- **判定あり**（`ホロウ_イベント`シートの該当選択肢の`判定難易度`が設定されている場合）: ホストが**パーティの中から1キャラを選び**、その1キャラの**探索判定**（既存の`events/socket_exploration.py::handle_exploration_roll`をそのまま流用。`nd6+選ばれたキャラのLv` vs `判定難易度`）に挑む。
  - 成功: 該当選択肢の`効果参照`の効果（小アイテム付与 or 一時バフ）を得る。
  - 失敗: `失敗時効果参照`の効果（状態異常/一時デバフ、既存の状態異常/デバフプラグインで表現。錯乱=confusion 等。新規効果タイプの追加は極力避ける）を、`失敗時対象範囲`（`self`=選んだキャラのみ／`party`=パーティ全体）に従って付与する。**対象範囲は選択肢ごとに変えてよい**（決定 → §9。環境系の澱みはparty全体、対話系の思念はselfのみ、といった描き分けを想定）。
- **判定なし**（`判定難易度`が空欄）: 判定を挟まず、該当選択肢の`効果参照`の効果を確定で得る（「やり過ごす」で`効果参照`も空欄＝ノーリスク・ノーリターン、という運用が基本形）。

**T4以降の一部の階**では、既存の**ステージ効果**（フィールド効果）を階の環境として自動設定する（例: 毎ラウンド出血圧・命中低下）。これは上記のイベント遭遇とは別に、階そのものの環境として常設される点に注意（イベント抽選で「魔素だまり」を引かなくても効果は継続する）。

### 4.7 パーティ編成画面（ダンジョンモード画面の一部）

調査の結果、「全味方キャラの中から一部を選び、順序を持たせる」概念はどこにも存在しない（`characters[]`は追加順に並ぶだけ。battle_onlyの編成モーダルはbattle_only専用）。ホロウのslot:N淵遺物には安定した編成順が必要なため新設するが、**§4.0の方針転換により、これは通常のTRPGルームに展開する汎用機能ではなく、ダンジョンモード画面専用のUI**として作る。

- **味方キャラ一覧**: このホロウ専用ルームの`characters[]`から`type=='ally'`を全件表示。味方キャラは各ユーザーが**アカウントの持ちキャラ（計画36）から選んで投入**する（正式経路。実装順序は§10 #11）。
- **パーティ一覧**: 味方キャラ一覧の**部分集合＋順序**。ルーム状態に`party.order`（`char_id`の配列）として新設し、`characters[]`本体は複製しない。
- **味方NPCの排除**: パーティ一覧に含めなければ、そのキャラは「味方だがパーティ外」として扱われる。これだけで排除は表現できるため、`is_npc`のような専用フラグは必須ではない（フラグを追加する場合はUI上の初期絞り込み・ラベル表示の補助用途に留める。§10で要否を確認）。
- **並べ替え**: パーティ一覧内でドラッグ or 上下ボタンによる順序変更。この順序が「先頭のキャラ」「2番目のキャラ」の基準になる。
- **ホロウとの連携**: ラン開始時に、その時点の`party.order`を`hollow.run.party_order`へ**スナップショット**する（ラン中にパーティ編成を変更しても、進行中のランのslot:N対象はランが終わるまで変わらない）。ランを開始するにはパーティが1人以上必要（空パーティでは開始不可）。
- **画面内の配置**: ダンジョンモード画面の常設パネル（階層ハブの一部）として表示する。通常ルームの`action_dock.js`には何も追加しない。

**パーティ画面を起点にした導線吸収**

パーティ一覧の各キャラ行に、既存機能への**ショートカット**を添える。新規に別の実装を作るのではなく、既存の呼び出し（`item_modal.js`のアイテム使用フロー、キャラクター詳細表示）をそのキャラID付きで起動するだけの薄いラッパーとする。

- 各キャラ行に🎒（アイテム使用）・詳細表示ボタンを配置し、押すと対象キャラを指定した状態で既存の`item_modal.js`/キャラ詳細モーダルを開く。**ショートカットはこの2つに限定する**（簡易ステータス編集📝は対象外。ステータス編集はGMが行う操作であり、パーティ画面から吸収する必要はない、と決定済み）。

### 5.1 ルーム状態への追加キー

```jsonc
// ルーム状態直下（このホロウ専用ルームのstate。データ構造自体は汎用だが、UI・導線はホロウ専用画面のみに作る）
"party": {
  "order": ["char_xxx", "char_yyy", "char_zzz"]   // 味方キャラ一覧の部分集合＋順序。ここに無い味方キャラは「パーティ外」
},

// ホロウ専用
"hollow": {
  "host_user_id": "user_xxx",   // ルーム単位で持続。既定値はRoomMemberのowner。ランをまたいで引き継がれる
  "run": {
    "status": "idle | in_progress | finished",
    "floor": 12,
    "party_order": ["char_xxx", "char_yyy", "char_zzz"],  // ラン開始時にparty.orderをスナップショット。slot:N淵遺物の基準
    "relics": [                            // パーティ共有。target_scopeで対象を解決する
      { "relic_id": "RL-001", "target_scope": "party" },
      { "relic_id": "RL-014", "target_scope": "slot:1" }
    ],
    // floors_left: 階数を1つ進むたびに1減る（イベント種別を問わない）。
    // battles_left: 戦闘が1回終わるたびに1減る（非戦闘イベントの階を通過しても減らない）。
    // 付与元の一時バフがどちらの単位で設定されているか（ホロウ_バフの「持続戦闘数」等）によって、
    // 該当する方のフィールドだけを持つ（両方持つ必要はない）→ §9
    "temp_buffs": [
      { "char_id": "char_xxx", "buff_id": "Bu-xx", "floors_left": 3 },
      { "char_id": "char_yyy", "buff_id": "HW-B-002", "battles_left": 2 }
    ],
    "current_event": {
      "type": "battle | elite | treasure | miasma | camp | boss",
      "resolved": false,
      "options": ["descend", "retreat"],
      "pins": { "descend": ["user_aaa"], "retreat": [] }   // 推薦ピン。非拘束・表示専用
    },
    "pending_drops": [],
    "seed_log": []           // 抽選結果の監査ログ（再現用ではなく記録用）
  },
  "records": { "best_floor": 34, "clear_count": 0, "boss_kills": ["10F", "20F"] }

```

- 真実はメモリ上の `active_room_states`、保存は既存デバウンスに乗せる（即時DB反映を前提にしない）。
- `pins` はホストの決定を待つ間だけ意味を持つ一時状態。イベント解決時にクリアしてよく、永続保存の必要性は低い（Phase 5で「保存するか／ephemeralな配信のみにするか」を実装時に決める）。
- `hollow.host_user_id` は初回アクセス時に未設定なら`Room.owner_id`で遅延初期化する。ホスト在室メンバーの退室（`RoomMember.revoked_at`打刻）を検知したら、`joined_at`昇順で次の在室メンバーへ再設定する（§4.1）。

### 5.2 マスターデータ（Google Sheets を正とする）

**Google Sheets を正**とし、既存のスキル/アイテム取り込みと同じ「Sheets → JSONキャッシュ（`data/cache/hollow_*_cache.json`）」パターンで取り込む。取り込みは既存の `--update` フロー（`manager/data_manager.py::update_all_data`）へフックを追加する。

- 読み取り方式は **gspread＋サービスアカウント**（`get_gspread_client` を流用）とする。公開CSV方式（items等）と違い非公開シートのままでよく、後述の書き戻しと同じ経路に統一できる。
- シートの置き場所は**ホロウ専用スプレッドシートを新設**（作成作業は進行中）。既存『ジェムリアTRPG_スキル一覧』とは分離し、書き戻し権限をホロウ側だけに絞れる（決定済み → §9）。**このスプレッドシートのID/共有先サービスアカウントへの編集権限付与は実装着手時に確定させる**。ワークシート単位で以下を設ける（2026-07-13、実装着手前に列レベルまで確定）。

### 5.2.1 Sheetsへの書き戻し（後続フェーズ）

サービスアカウントは gspread の書き込みAPIも使えるため、**対象シートに編集権限を付与すれば書き戻しは実現可能**（認証基盤は稼働実績あり）。ただしコア機能ではないため Phase 7 に分離する。

- 想定ユースケース: ①アプリ内エディタ（GM/管理者UI）で敵・淵遺物を編集→Sheetsへ書き戻し→再取り込み、②戦績・ドロップ実績の統計をレポート用ワークシートへ追記。
- 制約・設計方針: 書き込み対象は**ホロウ系ワークシートのみ**に限定（スキル等の既存シートには書かない）/ 行の特定は先頭列のIDで行い、手動編集との衝突は「Sheets側が正、書き戻しは上書き確認付き」で運用 / gspread APIのレート制限を考慮し書き込みはバッチ化 / 書き込み失敗時もアプリ動作（キャッシュ読み）には影響させない。

### Phase 0: 専用ルーム基盤・ロビー入口・パーティ/ホスト
