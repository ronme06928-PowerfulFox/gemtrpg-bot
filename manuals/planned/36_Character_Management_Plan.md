# 36 キャラクター管理基盤計画（持ちキャラ・キャラ作成統合・成長管理）

**作成日**: 2026-07-11
**最終更新**: 2026-07-11（§10の残り論点をすべて決定として確定。ホロウ由来セッションはアイテム反映対象外という原則も反映。未決定事項なし）
**位置づけ**: キャラ作成ツール（`CharaCreator/`）をダイスボット本体へ統合し、**アカウントごとの「持ちキャラ」**を作成・保存・管理できるようにする基盤計画。
持ちキャラは通常のTRPGルームへもホロウ（計画35）へも投入でき、シナリオクリア後の成長（経験値・スキル取得）をアプリ内で管理できるようにする。
計画35（ホロウダンジョン）の「味方キャラの持ち込み」はこの計画の持ちキャラ投入を正式経路とする（35はこの計画に依存）。

---

## 1. 目的

- キャラクターの作成〜保存〜ルーム投入〜成長のライフサイクルを、クリップボード手運びなしにアプリ内で完結させる。
- アカウント（`User`）に持ちキャラを紐づけ、ルームをまたいだキャラの再利用を可能にする（現状キャラは`Room.data`内にのみ存在しルーム従属）。
- シナリオ終了後の成長要素（経験値の蓄積、スキル追加、パラメータ上昇）を管理できるようにする。
- ホロウ（計画35）へ「シナリオで成長した持ちキャラを連れて行く」流れを実現する基盤を提供する。

## 2. 対象範囲

### 今回扱うもの

- 持ちキャラのDBテーブル（`OwnedCharacter`）とCRUD API。
- キャラ管理画面（一覧・詳細・削除・複製・JSONエクスポート/インポート）。
- CharaCreatorの本体からの配信（serve）と「アカウントに保存」導線の追加（最小統合）。
- 持ちキャラの「ルームへ投入」導線（通常ルーム・ホロウ両対応。投入はコピー）。
- セッション成果の書き戻し（経験値・アイテム等を明示操作で持ちキャラへ反映）。
- 成長の操作: 「キャラ作成ツールでの再編集」と「専用の軽量成長画面」の両方。

### 今回扱わないもの

| テーマ | 理由 |
|---|---|
| CharaCreatorのUI全面作り直し | 116KBの完成した単体アプリであり作り直しは大工事。まず最小統合（保存先の接続）で価値を出す（決定済み → §9） |
| ルーム内キャラと持ちキャラの常時自動同期 | デバウンス保存との整合が難しく、「試しに使ったら本体が傷つく」事故を招く。書き戻しは明示操作のみ（決定済み → §9） |
| 持ちキャラの他ユーザーへの譲渡・共有 | 所有モデルを単純に保つ。必要になったら別途検討 |
| `json_definition_builder.html`（JSON定義ビルダー）の統合 | スキル定義作成用の別ツール。キャラ管理とは無関係 |
| モバイル版対応 | mobile は開発停止方針（PC Web版中心） |

## 3. 現状整理

### 調査結果（2026-07-11）

| 項目 | 現状 |
|---|---|
| CharaCreator | `CharaCreator/GEMDICEBOT_CharaCreator.html`（約116KB）。完全に独立した単体HTML/JSアプリで、本体はserveしていない。マスターデータはGoogle Sheets（gviz/CSV）直結。出力はクリップボード/`.json`ダウンロードのみ |
| キャラJSONスキーマ | CharaCreator出力（`kind:"character", data:{name, status, params, commands, SPassive, inventory}`）は本体パーサ `static/js/common/char_json.js::parseCharacterJsonToCharacterData` がそのまま取り込める。**スキーマ互換性は良好、変換不要**（パーサ側でstatus→states分解・必須状態補填・ランタイムフィールド補完を吸収） |
| アカウント基盤 | `models.py::User`（UUID PK・login_name+password認証・`session['user_id']`）。稼働済み |
| ユーザー所有物のモデリング手本 | `models.py::RoomMember`（user_id FK・部分unique索引・論理削除）。`Room.owner_id` も同パターン |
| キャラの永続化 | **独立テーブルなし**。`Room.data`（JSON列）内の`state['characters']`にのみ存在しルーム従属。ルーム跨ぎの再利用は手動JSON持ち回りのみ |
| ルーム内キャラの所有者記録 | `events/socket_char.py::handle_add_character` が `char_data['owner_id'] = session['user_id']` を記録済み（L77-78）。所有権譲渡`handle_transfer_character_ownership`もあり |
| 類似のプリセット構造 | battle_onlyプリセットストア（`manager/battle_only_presets.py`、JSONファイル保存）はグローバル共有で**ユーザー所有の概念なし** |
| 成長要素 | **未実装**。CharaCreatorの「経験」「シナリオ経験」は作成時のスキル取得予算（`expLimit`）であって蓄積型ではない。ただしこの予算モデルは成長フローに発展させられる |

### 新規設計が必要なもの

| ID | 欠落 | 対応先 |
|---|---|---|
| C1 | ユーザー所有のキャラを永続保存するテーブル・APIがない | Phase 1 |
| C2 | 持ちキャラを一覧・管理する画面がない | Phase 1 |
| C3 | CharaCreatorが本体から配信されておらず、出力を直接保存する経路がない | Phase 2 |
| C4 | ルームへの投入導線が「JSON貼り付け」しかない（持ちキャラから選ぶUIがない） | Phase 3 |
| C5 | セッション成果（経験値・アイテム）を持ちキャラへ書き戻す仕組みがない | Phase 4 |
| C6 | 蓄積型の経験値と成長履歴のデータ構造がない | Phase 4 |
| C7 | 成長操作のUI（再編集導線・軽量成長画面）がない | Phase 5 |

## 4. 決定済み方針

| 論点 | 決定 | 意図 |
|---|---|---|
| CharaCreator統合の深さ | **最小統合（A案）**。既存HTMLをほぼそのまま本体からserveし、「アカウントに保存」ボタンと持ちキャラ読込を追加する | 完成済み単体アプリの作り直しを避け、保存先の接続だけで持ちキャラ化の価値を出す |
| 成長の操作方法 | **再編集（A）と専用成長画面（B）の両方を用意する** | 大きな作り直し（スキル構成の変更等）は再編集で、セッション後の小さな成長（スキル1つ追加等）は軽量画面で、と使い分ける |
| ルーム投入後の書き戻し | **投入はコピー（持ちキャラ本体は不変）。シナリオ終了時に明示操作で成果（経験値・アイテム等）を書き戻す（A案）** | 自動同期は整合性リスクが高い。「試しに使って死んでも本体は無事」がホロウとの相性も良い |
| 計画の位置づけ | **35（ホロウ）とは独立した基盤計画とし、35はこの計画に依存する** | キャラ管理は通常プレイにも効く汎用基盤。ホロウ固有の話と混ぜない |

## 5. 設計（たたき台）

### 5.1 DBモデル

```python
class OwnedCharacter(db.Model):
    __tablename__ = 'owned_characters'
    id = db.Column(db.String(36), primary_key=True)          # UUID
    user_id = db.Column(db.String(36), db.ForeignKey('users.id'), nullable=False, index=True)
    name = db.Column(db.String(255), nullable=False)          # 表示用（data内nameのコピー）
    data = db.Column(db.JSON, nullable=False)                 # キャラJSON本体（CharaCreator出力のdata部）
    exp_total = db.Column(db.Integer, default=0)              # 蓄積経験値（成長予算）
    growth_log = db.Column(db.JSON, default=list)             # 成長履歴 [{date, kind, detail}, ...]
    created_at / updated_at = db.Column(db.DateTime, ...)
    deleted_at = db.Column(db.DateTime, nullable=True)        # 論理削除（RoomMemberの流儀に合わせる）
```

- `data`は`Room.data`と同じJSON列パターン。キャラJSONをそのまま格納し、スキーマの細分化はしない。
- `exp_total`はCharaCreatorの「経験/シナリオ経験（作成予算）」を蓄積型に発展させたもの。再編集時の`expLimit`再計算の入力になる。
- `growth_log`は成果反映・成長操作の監査ログ（いつ・何を得たか）。

### 5.2 API / Socket

| エンドポイント | 内容 |
|---|---|
| `GET /api/owned_characters` | 自分の持ちキャラ一覧（要ログイン） |
| `POST /api/owned_characters` | 新規保存（CharaCreatorから/JSONインポートから） |
| `PUT /api/owned_characters/<id>` | 更新（再編集の保存・成長画面の確定） |
| `DELETE /api/owned_characters/<id>` | 論理削除 |
| `request_add_character` 拡張 | 既存のルーム投入Socketに「持ちキャラID指定」を追加。サーバ側で`OwnedCharacter.data`をdeepcopyして投入し、`char_data['owned_character_id']`を記録（書き戻し先の対応付け） |
| `request_reflect_session_results`（新設） | セッション成果の書き戻し。対象キャラ（`owned_character_id`を持つルーム内キャラ）ごとに経験値・アイテムを持ちキャラへ反映し`growth_log`へ記録。**ホロウ（計画35）由来のセッションでは経験値のみを反映対象とし、アイテムは対象外**（計画35の決定「ホロウ内で完結の原則」に合わせる。通常のTRPGルームのセッションでは経験値・アイテムとも反映対象） |

- 認可: 持ちキャラCRUDは所有者本人のみ。成果反映の実行権限は**キャラ所有者本人またはGM/ホスト**（決定 → §9）。

### 5.3 CharaCreator統合（最小統合）

- `CharaCreator/GEMDICEBOT_CharaCreator.html` を本体ルート（例: `/chara_creator`）からserveする。既存のクリップボード/ダウンロード機能は**そのまま残す**（従来運用の互換維持）。
- 追加するのは2点のみ:
  1. **「アカウントに保存」ボタン**: 現在の`updateJSON()`出力を`POST /api/owned_characters`へ送る（ログイン済みセッションのcookieを利用）。
  2. **持ちキャラ読込**: URLパラメータ or 選択UIで`OwnedCharacter`を読み込み、フォームへ復元（再編集の入口。`exp_total`から`expLimit`を再計算して成長分の予算を反映）。
- マスターデータ取得は当面Sheets直結（gviz）のまま。将来的に本体キャッシュ経由へ切り替えるかは§10。

### 5.4 キャラ管理画面

- ロビー画面からアクセスできる「マイキャラクター」画面（ホロウ入口と同様、ロビー起点の導線）。
- 機能: 一覧（名前・更新日・exp_total）／詳細表示（既存キャラ詳細モーダルの流用）／削除・複製／JSONエクスポート（従来運用との互換）／JSONインポート（手持ちのキャラJSONを持ちキャラ化）／「キャラ作成ツールで編集」「成長画面」への導線。

### 5.5 ルーム投入と成果反映

- **投入**: キャラ追加導線（計画28 P1-4の2段階ウィザード）に「持ちキャラから選ぶ」タブを追加。投入は**コピー**であり、ルーム内での変化は持ちキャラ本体に影響しない。ホロウ（計画35）のダンジョンモード画面でも同じ投入APIを使う。
- **成果反映**: セッション終了時（またはGMの任意タイミング）に「成果を反映」操作を実行。対象キャラの経験値付与・アイテム反映を選び、`owned_character_id`経由で持ちキャラへ書き戻す。反映内容は`growth_log`に記録。
- ルーム内キャラに`owned_character_id`が無い場合（従来のJSON貼り付けキャラ）は反映対象外（従来どおり）。

### 5.6 成長の操作（2系統）

| 系統 | 用途 | 実装 |
|---|---|---|
| A: 再編集 | スキル構成の組み直し等、大きな変更 | CharaCreatorに持ちキャラを読み込み、`exp_total`から`expLimit`を再計算した状態で編集→上書き保存 |
| B: 軽量成長画面 | セッション後の小さな成長（スキル1つ追加・パラメータ+1等） | キャラ管理画面内の専用UI。残り経験値（`exp_total` − 使用済み）の範囲でスキル追加・パラメータ上昇を選び、`data`へ反映して`growth_log`に記録 |

- 両系統とも「使用済み経験値」の計算方法を共通化する（データはCharaCreatorの取得コスト相場に従う。B画面のコスト計算はCharaCreatorのロジックを参照実装とする）。

## 6. 実装フェーズ

### Phase 1: 持ちキャラ基盤（テーブル・API・管理画面の最小形）

- `OwnedCharacter`テーブルとマイグレーション（`manager/db_migration.py`の流儀に従う）。
- CRUD API（一覧/保存/更新/削除、所有者認可）。
- キャラ管理画面の最小形（一覧・削除・JSONインポート/エクスポート）。ロビーからの導線。
- テスト: CRUD・認可（他人のキャラは見えない/触れない）・論理削除。

### Phase 2: CharaCreator統合（最小統合）

- `/chara_creator`ルートでのserve（静的配信）。
- 「アカウントに保存」ボタン（`POST /api/owned_characters`）。
- 持ちキャラ読込（再編集の入口。まずは読込→編集→上書き保存の一巡）。
- テスト: 保存APIの入力検証（`parseCharacterJsonToCharacterData`相当の正規化をサーバ側でも実施）。

### Phase 3: ルーム投入導線

- `request_add_character`の持ちキャラID対応（サーバ側deepcopy投入＋`owned_character_id`記録）。
- キャラ追加ウィザードへの「持ちキャラから選ぶ」タブ追加（通常ルーム）。
- ホロウ（計画35）のパーティ編成パネルからの投入接続は35側Phase 0と調整。
- テスト: 投入コピーの独立性（ルーム内で変更しても本体不変）、owned_character_idの記録。

### Phase 4: 成果反映と経験値蓄積

- `request_reflect_session_results`: 経験値付与・アイテム反映（通常ルームのみ）・`growth_log`記録。呼び出し元がホロウ（計画35）かどうかでアイテム反映の有無を切り替える。
- `exp_total`の蓄積と「使用済み経験値」の算出（データから毎回逆算する方式。決定 → §9）。
- テスト: 反映の冪等性（二重反映防止）、対象外キャラ（owned_character_idなし）のスキップ、ホロウ由来セッションでアイテムが反映されないことの確認。

### Phase 5: 成長UI（軽量成長画面＋再編集の仕上げ）

- 軽量成長画面（スキル追加・パラメータ上昇、残り経験値の範囲チェック）。
- CharaCreator再編集時の`expLimit`再計算（`exp_total`連動）。
- テスト: コスト計算の整合（CharaCreatorのロジックと同値になること）。

## 8. 受け入れ条件

| フェーズ | 条件 |
|---|---|
| Phase 1 | ログインユーザーが自分の持ちキャラを保存・一覧・削除でき、他人のキャラには一切アクセスできない |
| Phase 2 | CharaCreatorで作ったキャラを画面遷移なしにアカウントへ保存でき、保存済みキャラを読み込んで再編集→上書きできる |
| Phase 3 | 持ちキャラをルームへ投入でき、ルーム内での変化が本体に影響しないことをテストで確認できる |
| Phase 4 | セッション成果の反映操作で経験値（通常ルームはアイテムも）が持ちキャラへ書き戻され、`growth_log`に履歴が残る。二重反映が防止される。ホロウ由来ではアイテムが反映されないことをテストで確認できる |
| Phase 5 | 軽量成長画面での成長がコスト計算どおりに動き、再編集でも同じ経験値予算が使われる |
| 共通 | `pytest -q` 全通過、`npm run build` 済み成果物コミット、正本を系統ファイルへ統合して本計画書を削除 |
