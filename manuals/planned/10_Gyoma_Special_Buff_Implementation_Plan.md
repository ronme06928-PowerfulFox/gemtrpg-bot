# 10 凝魔・蓄力（双リソース特殊バフ）実装計画

**作成日**: 2026-03-29  
**最終更新日**: 2026-04-08  
**対象**: 新規特殊バフ `凝魔` / `蓄力`  
**位置づけ**: 実装前の仕様確定用計画書

---

## 1. 目的

`凝魔` と `蓄力` を、同一の仕組みでスタック・消費できる特殊バフとして導入する。  
両者は名称と用途を分離し、以下を実現する。

- `凝魔`: 魔法系の追加効果・補正に使用
- `蓄力`: 物理系の追加効果・補正に使用

加えて、両バフに共通で「10スタックごとに補正+1」を適用する。

---

## 2. 現状実装への適合性（実情ベース）

現行コードには次の土台が既にあるため、拡張で対応可能。

1. `special_buffs` に `count` を持つ永続バフの加算処理が存在する
- `manager/utils.py::apply_buff`
- 既存の `Bu-08`, `Bu-11`, `Bu-12` で count 管理の前例あり

2. 永続バフ（`is_permanent=true` / `lasting=-1`）のラウンド維持処理が存在する
- `manager/battle/common_manager.py`

3. `物理補正` / `魔法補正` は既にダメージ算出で参照される
- `manager/utils.py::get_buff_stat_mod`
- `manager/game_logic.py` の `{物理補正}` / `{魔法補正}` 展開

4. 右上バッジ表示は現状 `states` ベース
- `static/js/visual/visual_map.js::generateMapTokenBadgesHTML`
- 特殊バフ表示には追加実装が必要

---

## 3. 仕様要件（今回追加分を反映）

1. `凝魔` と `蓄力` は同様にスタックする別名称バフ
2. 効果で消費する時は、対象バフ名を明示して処理
3. 魔法/物理の区別はリソース名で分離
4. `凝魔` は 10スタックごとに `魔法補正 +1`
5. `蓄力` は 10スタックごとに `物理補正 +1`

補正式:

- `gyoma_bonus = floor(凝魔.count / 10)`
- `chikuryoku_bonus = floor(蓄力.count / 10)`

---

## 4. データモデル

## 4.1 バフ定義

- 凝魔
  - `name`: `凝魔`
  - `buff_class`: `special_resource`
  - `display_group`: `buff`
  - `is_negative`: `false`
- 蓄力
  - `name`: `蓄力`
  - `buff_class`: `special_resource`
  - `display_group`: `buff`
  - `is_negative`: `false`

推奨内部ID（案）:

- `Bu-Gyoma`
- `Bu-Chikuryoku`

## 4.2 共通管理項目

- `count`: 残りスタック（消費成功で減少）
- `lasting`: 継続ラウンド
- `delay`: 有効化遅延（通常0）
- `is_permanent`: 永続フラグ

既定値ルール:

- `lasting` 未指定時は `-1`（無制限）
- 明示指定時のみ有限ラウンド運用

## 4.3 固定説明文

- 凝魔: `特定のスキル効果によって獲得する。残り回数を消費し、一部スキルの追加効果を適用することができる。`
- 蓄力: `特定のスキル効果によって獲得する。残り回数を消費し、一部スキルの追加効果を適用することができる。`

※文面は同一、名称のみ置換して表示。

## 4.4 既存5状態異常の分類固定（回帰防止）

既存5状態異常（`出血`・`破裂`・`亀裂`・`戦慄`・`荊棘`）は、従来どおり次で固定する。

| 状態異常 | status_class | display_group | is_negative |
| --- | --- | --- | --- |
| 出血 | `debuff` | `debuff` | `true` |
| 破裂 | `debuff` | `debuff` | `true` |
| 亀裂 | `debuff` | `debuff` | `true` |
| 戦慄 | `debuff` | `debuff` | `true` |
| 荊棘 | `debuff` | `debuff` | `true` |

## 5. 効果定義

## 5.1 獲得

`APPLY_BUFF` を使用し、`buff_name` で `凝魔` または `蓄力` を付与する。  
既存同名バフがある場合は `count` を加算する。

## 5.2 消費して効果獲得

共通効果（案）:

- `CONSUME_BUFF_COUNT_FOR_GAIN`
- 必須:
  - `buff_name` (`凝魔` / `蓄力`)
  - `consume_required`
  - `gains`

判定:

- `current_count >= consume_required` のときのみ成功
- 不足時は不発（消費なし・獲得なし）

## 5.3 10スタック補正

補正は `count` から動的計算する。

- `凝魔`: `floor(count / 10)` を `魔法補正` に加算
- `蓄力`: `floor(count / 10)` を `物理補正` に加算

実装推奨:

- `manager/utils.py::get_buff_stat_mod` と `get_buff_stat_mod_details` で、`凝魔` / `蓄力` を特例計算
- `stat_mods` の事前同期ではなく、参照時算出にする（count更新漏れ回避）

---

## 6. 魔法/物理スキルの区別方針

本計画では、スキル種別の複雑な新判定は導入しない。  
既存の `物理補正` / `魔法補正` 参照系をそのまま利用する。

- 魔法系スキル: コマンド式が `{魔法補正}` を参照
- 物理系スキル: コマンド式が `{物理補正}` を参照

これにより、`凝魔` と `蓄力` の補正は自然に分離適用される。

---

## 7. ラウンド管理

- `lasting=-1`（未指定既定）は無制限継続
- `lasting>0` の場合のみラウンド終了で減衰
- `count<=0` で即時除去

減衰・除去の扱いは、既存の亀裂関連バフのラウンド処理フローに合わせる。

---

## 8. UI表示（右上バッジ）

現状の右上バッジは `states` のみを描画しているため、以下を追加する。

1. `generateMapTokenBadgesHTML` に `special_buffs` 表示分岐を追加
2. 対象バフ: `凝魔`, `蓄力`
3. 表示内容: アイコン + `x<count>`
4. ツールチップ: 固定説明文 + 現在スタック + 10スタック補正値

補足:

- アイコン画像が未用意の間は、テキストバッジ（凝/力）でフォールバック

## 9. 実装フェーズ

## Phase A: 定義同期

- Buff Catalog に `凝魔` / `蓄力` を追加
- 用語定義・説明文・分類メタを反映
- 既存スキル定義の `蓄魔` が残っていれば `凝魔` へ置換

- `apply_buff` に `凝魔` / `蓄力` のスタック加算・既定永続を追加

- `get_buff_stat_mod` / `get_buff_stat_mod_details` に10スタック補正計算を追加

- 駒右上に `凝魔` / `蓄力` バッジ表示
- ツールチップに説明文と補正値を表示

## 10. テスト観点

1. `凝魔` / `蓄力` が `special_buffs` に正しく加算される
2. `lasting` 未指定時に無制限継続する
3. `consume_required` 以上でのみ消費成功し、`gains` が適用される
4. `consume_required` 超過要求時は不発（消費なし・獲得なし）
5. `凝魔` の補正が `count=0..9` で +0、`10..19` で +1、`20..29` で +2 になる
6. `蓄力` も同様に10刻みで補正が上がる
7. `{魔法補正}` 参照スキルでのみ凝魔補正が効く
8. `{物理補正}` 参照スキルでのみ蓄力補正が効く
9. 右上バッジに `凝魔 xN` / `蓄力 xN` が表示される
10. 既存5状態異常が `debuff / debuff / true` を維持する

---

## 11. 受け入れ条件

1. `凝魔` と `蓄力` が別名称の特殊バフとして定義されている
2. 両者が同方式でスタック・消費できる
3. 10スタックごとの補正ルールが実装・検証されている
4. `凝魔` は魔法補正、`蓄力` は物理補正にのみ寄与する
5. 不足時不発ルール（消費なし・獲得なし）が保証される
6. 駒右上表示と説明表示が反映される
