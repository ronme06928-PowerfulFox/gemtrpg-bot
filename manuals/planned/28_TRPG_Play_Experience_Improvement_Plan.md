# 28 TRPGプレイ体験改善計画

**作成日**: 2026-07-07
**最終更新**: 2026-07-10（Phase 1実装完了。正本は `manuals/implemented/F01_Operations_Manual.md` Part 7）
**位置づけ**: アプリ全体を「TRPGをより安定して、扱いやすく、記録しやすくする」ための改善計画。戦闘ルールや個別スキル追加ではなく、セッション運営・保存信頼性・ログ・通知・検索・アクセシビリティなどのプレイ体験基盤を対象にする。

---

## 1. 目的

- セッションの記録を失わない。ログ、ダイス、保存処理を信頼できる状態にする。
- GMの運営負荷を下げる。通知、イニシアチブ調整、限定undo、操作履歴を整える。
- 長時間プレイの快適性を上げる。ショートカット、ダークモード、検索、最低限のアクセシビリティを整える。
- 実装済みになった古い計画や死にコードを整理し、今後の実装対象を明確にする。

## 2. 対象範囲

### 今回扱うもの

- 保存失敗時の再試行と節目フラッシュ。
- セッションログの長期保全、エクスポート、検索。
- `/roll` `/sroll` の実装済みサーバーダイス仕様の正本化と回帰テスト。
- 手番・宣言待ち通知。
- イニシアチブのGM手動調整。
- ステータス変更に限った操作履歴ベースのundo。
- キーボードショートカット。
- 主要画面からの段階的ダークモード。
- 可視性制御付きスキル検索の再設計。
- 主要操作ボタンの `aria-label` 付与。
- ルート直下などに残る開発残骸ファイルの整理。

### 今回扱わないもの

以下は単独で大きい設計になるため、本計画では候補として残し、着手時に個別計画へ切り出す。

| テーマ | 理由 |
|---|---|
| 観戦者ロール | secret/gmOnly 情報の公開範囲、UI操作制限、入室権限の再設計が必要 |
| 並行戦闘 | `battle_id = f"battle_{room}"` 前提を外す大きな状態構造変更が必要 |
| シナリオ/キャンペーン管理 | ルーム状態、ハンドアウト、進行メモの新しい永続化設計が必要 |
| BGM/効果音同期 | 音源権利、ホスティング、同期再生制御を別途詰める必要がある |
| VTT拡張 | フォグ、ピン、メモ、複数マップなど探索モード全体の設計が必要 |

また、逆襲、気合い、増援、キャラ駒の枠画像などは既存の個別計画で扱う。

## 4. 決定済み方針

| 論点 | 決定 | 意図 |
|---|---|---|
| ログ長期保全 | 切り捨て前に別テーブルへアーカイブする。 | 500件超過後もセッション記録を失わない。GMの手動保存忘れにも耐える。 |
| 04残課題との統合 | GM操作ログ見える化を限定undoへ吸収する。 | 操作履歴データを二重に作らず、undoと監査表示を同じ基盤で扱う。 |
| undoの範囲 | HP/MP/FP・状態異常などのステータス変更の逆操作のみ。 | スナップショット復元は召喚、バフ期限、戦闘メモリとの整合が重いため、まず日常的な誤操作救済に絞る。 |
| サーバーダイス仕様 | A01にユーザー向け仕様、F01に信頼性/運用仕様を記載し、回帰テストを追加する。 | 実装済み挙動を正本化し、将来の退行を防ぐ。 |
| スキル検索 | 図鑑/用語UIへ統合し、非GMは敵専用スキルを検索対象外にする。ただし現在フィールドに存在するキャラクター詳細モーダルで閲覧可能なスキルは検索可能にする。 | 全件検索によるネタバレを避けつつ、今見えている敵の公開済みスキルは探せるようにする。 |
| スキル検索UI | 旧 `tab_skill_search.js` の単純復活ではなく、現在のアプリの雰囲気に合わせた検索モーダルとして再設計する。 | 復活機能だけが古い見た目・操作感で浮くことを避ける。 |
| ダークモード | 主要画面から段階導入する。 | 全画面一括は確認範囲が広く崩れやすいため、戦闘画面、ログ、モーダルから実用価値を出す。 |
| P3大型テーマ | 今回は着手しない。必要時に個別計画へ切り出す。 | 28の主目的である信頼性、運営支援、快適性改善から分離する。 |

## 5. 実装フェーズ

### Phase 1: 保存信頼性とログ保全（実装完了）

2026-07-10実装完了。正本は `manuals/implemented/F01_Operations_Manual.md` Part 7。

#### P0-3 保存信頼性

- デバウンス保存失敗時の1回再試行を実装済み。
- ラウンド開始/終了、戦闘リセット、強制マッチ終了、戦闘モード切替、探索/戦闘パート切替で即時フラッシュを実装済み。
- 削除済みルームを再試行保存で復活させないガードを実装済み。

#### P0-1 ログ別テーブルアーカイブ、エクスポート、検索

- `room_log_archives` テーブルへの切り捨て前アーカイブを実装済み。
- GM向け `GET /api/room/export_logs` を実装済み。JSON/TXTに対応し、現行ログとアーカイブログを統合する。
- ログ履歴モーダルのフリーワード検索と種別フィルタを実装済み。
- GM向けログ履歴モーダルにJSON/TXTエクスポートボタンを追加済み。

### Phase 1.5: サーバーダイス仕様の固定

- `/roll` の通常公開ログ、`/sroll` の秘匿ログを回帰テストで固定する。
- 投稿者名はクライアント入力ではなく、サーバー側の在室情報から確定することをテストする。
- 不正なダイス式、ダイス式なし、複数コマンド混在時の扱いを仕様化する。
- `manuals/implemented/A01_User_Guide_Core.md` には利用者向けの書式と表示を記載する。
- `manuals/implemented/F01_Operations_Reliability.md` にはサーバー生成、秘匿、監査性、退行テスト観点を記載する。

### Phase 2: 通知とショートカット

#### P1-1 手番・宣言待ち通知

- 自分の手番、宣言待ち、解決待ちになったときにタブタイトルを点滅させる。
- ブラウザ `Notification` はユーザー許可済みの場合だけ使う。
- 宣言待ちSEは既存の `sound_fx.js` とSEトグルに従う。
- サーバー変更なしを基本とし、`BattleStore` の状態購読で完結させる。

#### P2-1 キーボードショートカット

- 宣言確定、解決フェーズ送り、ズームリセット、チャット欄フォーカスなど、主要操作だけに絞る。
- 入力欄、テキストエリア、モーダル内フォームにフォーカスがある場合は発火させない。
- ダイアログのEsc/Enterは既存の操作と衝突しないように対象を限定する。

### Phase 3: GM運営支援

#### P1-2 イニシアチブ手動調整

- GM限定イベントでタイムライン順序を変更できるようにする。
- UIは現行タイムラインに合わせ、まずは上下ボタンまたはシンプルな並べ替え操作を優先する。ドラッグ&ドロップは必要なら後続でよい。
- 変更後の順序は全参加者へ同期する。
- 存在しないキャラクター、重複、対象外スロットの混入をサーバー側で拒否する。

#### P1-3 操作履歴ベースの限定undo

- HP/MP/FP・状態異常などのステータス変更を、操作履歴として直近N件保持する。
- 履歴には操作ID、操作者、対象キャラクター、変更項目、変更前、変更後、時刻を残す。
- GMは直近履歴から逆操作で戻せる。
- 04残課題の「GM操作ログ見える化」は、この履歴をGM向けに表示することで吸収する。
- 戦闘フェーズ全体のスナップショット復元、召喚・バフ期限・戦闘メモリの巻き戻しは今回対象外。

### Phase 4: 快適性と検索

#### P2-2 主要画面ダークモード

- CSS変数化を先に進め、戦闘画面、ログ、モーダルから段階導入する。
- `prefers-color-scheme: dark` と手動トグルを用意し、手動設定は `localStorage` に保存する。
- 機能変更は入れず、コントラスト、可読性、既存の雰囲気との整合を優先する。

#### P2-3 可視性制御付きスキル検索

- 旧 `tab_skill_search.js` を全件検索UIとして単純復活させない。
- 図鑑/用語UIへ統合した検索モーダルとして作り直す。
- UIは現在のアプリの雰囲気に合わせ、既存モーダル、ドック、キャラクター詳細の操作感から浮かないようにする。
- 検索結果は密度高めにし、分類、距離、タグ、所持者/表示元などで絞り込めるようにする。
- 詳細表示は既存の `openSkillDetailModal` と整合させる。
- GMは全スキル検索可能。
- 非GMは、敵専用ではない公開スキルと、現在フィールド上のキャラクター詳細モーダルで閲覧可能なスキルだけ検索可能。
- 現行の `/api/get_skill_data` は全スキルマスタを返すため、検索UIだけのフロントフィルタでは不十分。サーバー側で閲覧可能スキル集合を返すAPIへ置き換えるか、既存APIを権限制御付きに変更する。

#### P2-4 アクセシビリティ最低限対応

- ドックアイコンと主要操作ボタンへ `aria-label` を付与する。
- アイコンのみのボタンは、必要に応じてツールチップも合わせる。
- P2-1のキーボード操作と合わせ、最低限の操作可能性を底上げする。

### Phase 5: 残骸ファイル整理

- `error.log`、`test_results.txt`、`temp_animation.css`、`manager/temp_logic_draft.py` など、ルート直下や一部ディレクトリの開発残骸を確認して削除する。
- 削除前に参照がないことを `rg` で確認する。
- 実ログやユーザー作成物である可能性があるものは、即削除せず用途確認または別途退避方針を決める。
