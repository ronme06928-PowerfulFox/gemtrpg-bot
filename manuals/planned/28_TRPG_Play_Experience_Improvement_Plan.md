# 28 TRPGプレイ体験改善計画

**作成日**: 2026-07-07
**最終更新**: 2026-07-11（計画28を一旦完了扱いでクローズ）
**位置づけ**: アプリ全体を「TRPGをより安定して、扱いやすく、記録しやすくする」ための改善計画。Phase 1/1.5/P1-4の実装完了と、残候補の対象外判断により一旦クローズする。

---

## 1. 目的

- セッションの記録を失わない。ログ、ダイス、保存処理を信頼できる状態にする。
- 実装済みになった保存信頼性、ログ保全、サーバーダイス、キャラクター追加導線を正本化する。
- 現行UIで必要性が低い改善候補を残タスクから外し、28のスコープを閉じる。
- 将来必要になった大きなテーマは、28へ積み戻さず個別計画として再検討できるようにする。

## 2. 対象範囲

### 今回扱うもの

- 保存失敗時の再試行と節目フラッシュ。
- セッションログの長期保全、エクスポート、検索。
- `/roll` `/sroll` の実装済みサーバーダイス仕様の正本化と回帰テスト。
- キャラクターJSON追加導線の再設計。

### 今回扱わないもの

以下は単独で大きい設計になるため、本計画では候補として残し、着手時に個別計画へ切り出す。

| テーマ | 理由 |
|---|---|
| 観戦者ロール | secret/gmOnly 情報の公開範囲、UI操作制限、入室権限の再設計が必要 |
| 並行戦闘 | `battle_id = f"battle_{room}"` 前提を外す大きな状態構造変更が必要 |
| シナリオ/キャンペーン管理 | ルーム状態、ハンドアウト、進行メモの新しい永続化設計が必要 |
| BGM/効果音同期 | 音源権利、ホスティング、同期再生制御を別途詰める必要がある |
| VTT拡張 | フォグ、ピン、メモ、複数マップなど探索モード全体の設計が必要 |
| 通知/キーボードショートカット | 現行UIではスロットと解決画面で宣言・解決状態を把握でき、追加通知やショートカットの必要性が低い |
| イニシアチブ手動調整 | 現行UIにタイムライン前提がなく、必要性が低い |
| 限定undo/操作履歴 | 現時点では必要性が低く、保存・戦闘状態との整合コストが上回る |
| ダークモード | 使い勝手上の強い課題が出た時点で別途検討する |
| 可視性制御付きスキル検索 | 検索導線の必要性が高まった時点で、ネタバレ制御を含む個別計画として再検討する |
| `aria-label` / 残骸整理 | 単独の28残タスクにはせず、関連UI改修や整理作業の中で必要に応じて扱う |

また、逆襲、気合い、増援、キャラ駒の枠画像などは既存の個別計画で扱う。

## 5. 実装フェーズ

### Phase 1: 保存信頼性とログ保全（実装完了）

2026-07-10実装完了。正本は `manuals/implemented/F01_Operations_Manual.md` Part 7。

- デバウンス保存失敗時の1回再試行を実装済み。
- ラウンド開始/終了、戦闘リセット、強制マッチ終了、戦闘モード切替、探索/戦闘パート切替で即時フラッシュを実装済み。
- 削除済みルームを再試行保存で復活させないガードを実装済み。

#### P0-1 ログ別テーブルアーカイブ、エクスポート、検索

- `room_log_archives` テーブルへの切り捨て前アーカイブを実装済み。
- GM向け `GET /api/room/export_logs` を実装済み。JSON/TXTに対応し、現行ログとアーカイブログを統合する。
- ログ履歴モーダルのフリーワード検索と種別フィルタを実装済み。
- GM向けログ履歴モーダルにJSON/TXTエクスポートボタンを追加済み。

### Phase 1.5: サーバーダイス仕様の固定（実装完了）

2026-07-11実装完了。正本は `manuals/implemented/A01_Player_Manual.md` と `manuals/implemented/F01_Operations_Manual.md` Part 8。

- `/roll` の通常公開ログ、`/sroll` の秘匿ログを回帰テストで固定済み。
- 投稿者名はクライアント入力ではなく、サーバー側の在室情報から確定することをテスト済み。
- 不正なダイス式、ダイス式なし、複数コマンド混在時の扱いを仕様化済み。
- `/roll` はpayloadの `secret` に影響されず公開ログ、`/sroll` は秘匿ログとして扱う。
- 先頭の独立トークンだけをコマンド扱いし、複数コマンド混在時は分割実行しない。

### Phase 2: 廃止

2026-07-11に廃止決定。通知、視認性強調、キーボードショートカットは実装しない。

理由:

- 現行のビジュアル戦闘は「自分の手番が来る」構造ではなく、スロットとSelect/Resolveで宣言状態を把握する。
- 解決開始は既にボタンの有効/無効で判定できる。
- マッチ中は専用の解決画面が通るため、追加のマップ上強調や状態表示の価値が低い。
- ログやスロットが大きなボトルネックではない現状で、通知/ショートカット/表示追加を入れるとUIノイズと保守対象だけが増えやすい。

既に実装済みのP1-4（キャラクターJSON追加導線）は、Phase 2廃止後も完了済みタスクとして保持する。

#### P1-4 キャラクターJSON追加導線の再設計（実装完了）

2026-07-10実装完了。正本は `manuals/implemented/E02_UI_Component_Specs.md` Part 5。

現行の `openCharLoadModal` は、JSON貼り付けフォームと「味方として追加」「敵として追加」を同一画面に並べている。これを、追加先選択とJSON入力を分離した2段階ウィザードへ変更する。

推奨フロー:

1. 「キャラクター追加」ボタンを押す。
2. 最初のモーダルで「味方を追加」「敵を追加」を選ぶ。
   - 「味方を追加」は通常プレイヤーも利用可能。
   - 「敵を追加」も現時点では通常プレイヤーに開放する。GM限定化は後続判断とする。
3. 追加先を選ぶと、JSON入力モーダルへ進む。
4. JSON入力モーダルには次を並べる。
   - JSON貼り付け用テキストエリア。
   - 「ファイルから読み込み」ボタン。
   - 非表示の `<input type="file" accept=".json,application/json">`。
   - 「追加」ボタン。
5. 「ファイルから読み込み」を押すとローカルJSONファイルを選択し、`FileReader` で読み込んだ内容をテキストエリアへ反映する。
6. 「追加」ボタンは、テキストエリアの内容を既存の `loadCharacterFromJSON(type, jsonText, resultMsg)` へ渡す。
7. 貼り付け入力の場合も、ファイル読込の場合も、追加処理は同じ検証・同じSocket送信経路を通す。

UI方針:

- 追加先の選択、入力方式、最終追加を分け、誤って敵として入れる事故を減らす。
- ファイル読込後はファイル名と読込成功メッセージを表示する。
- JSON解析に失敗した場合は追加前にエラーを表示し、モーダルを閉じない。
- 既存のデバッグキャラ生成は、各追加画面内のGM専用上級操作として残す。ただし通常導線とは視覚的に分離する。
- GM専用デバッグキャラ生成には追加人数のプルダウンを付ける。味方追加画面では「デバッグ味方を生成」、敵追加画面では「デバッグ敵を生成」として、その画面で選んだ陣営にだけ生成する。
- 追加人数はまず少数に限定する（例: 1、2、3、4、5体）。大量生成によるUI崩れや保存負荷を避けるため、上限を設け、必要なら後続で調整する。
- ローカルファイルはブラウザで読み込むだけで、ファイルパスやファイル本体をサーバーへ直接送らない。サーバーへ送るのは既存どおりJSON文字列から生成されたキャラクターデータのみ。

実装対象:

- `static/js/modals.js::openCharLoadModal`
- `static/js/common/char_json.js` の既存パース関数との整合確認
- 通常の敵JSON追加は既存どおり非GMにも開放する
- `request_add_debug_character` は、指定人数ぶん生成できるようにする。サーバー側でもGM限定、人数上限、整数検証を行う。
- JS変更後は `npm run build`
