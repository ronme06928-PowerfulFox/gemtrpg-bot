# 37 キャラクタータグ・条件判定・シナリオギミック拡張計画

**作成日**: 2026-07-20
**最終更新**: 2026-07-20
**状態**: PR 1実装済み・PR 2〜3着手可能（PR 4の内部形式とシナリオデータIDは未決定）
**主要ユースケース**: シナリオ No.2『瓦礫で鋳るは魔の鋼』のボススキル「鉄塊を捥ぎり喰う」

---

## 1. 目的

キャラクターを名前以外の固定分類で識別できるタグ基盤を追加し、スキル条件、PvE自動対象選択、死亡時効果から共通利用できるようにする。

同時に、対象の最大HPを参照する汎用ダメージ、文字列condition、死亡原因となったスキル使用者の引き継ぎ、順序付きPvE対象候補を実装する。「鉄塊を捥ぎり喰う」はこれらの汎用機能を組み合わせて定義し、スキル名に依存した専用処理は作らない。

---

## 2. 対象範囲

### 2.1 今回扱うもの

- キャラクターJSONの`tag_ids: string[]`と`disabled_tag_ids: string[]`。
- プレイヤーキャラクターへの出身国タグの自動付与。
- 輝化スキルによるタグ取得、通過点消費、解除時の通過点返還。
- GMがNPC・エネミーを作成するときの自由タグ入力。
- キャラクタービルダーでのタグ表示・編集・有効状態の権限分離。
- キャラクター詳細画面の名前直下へのタグ一覧表示。
- `condition`での文字列完全一致・部分一致・タグ包含判定。
- 汎用`CUSTOM_EFFECT`である`DEAL_TARGET_MAX_HP_DAMAGE`。
- ダメージ発生元から死亡時パッシブまでの使用者・スキル情報の引き継ぎ。
- HPが正数から0になったときだけ死亡時効果を実行する制御。
- PvE行動フローチャートの順序付き対象候補。
- `target_scope: "any"`を使う同陣営ギミック対象と敵陣営フォールバックの共存。
- 「鉄塊を捥ぎり喰う」と瓦礫死亡時パッシブによる結合確認。

### 2.2 今回扱わないもの

- プレイヤーによる自由入力タグ。
- タグそのものへ独自の効果実行エンジンを持たせること。
- 「鉄塊を捥ぎり喰う」というスキル名に紐づくハードコード。
- シナリオ本文、敵HP、戦闘バランスの最終調整。
- キャラクター名をタグへ自動変換する推測処理。
- 独立したタグマスター用スプレッドシート。
- タグ自体に説明文、色、効果などのメタデータを持たせること。
- 瓦礫も生存PCも存在しない戦闘状態への専用フォールバック。生存PCがいない場合は通常の敗北処理へ委ねる。

### 2.3 既存機能として維持するもの

- 輝化スキルの取得コストは通過点を上限として計算する。
- PvEの行動内容は行動フローチャートが決定する。
- スキルの対象可能範囲は`target_scope`が決定する。
- プレイヤーがより高速な行動でボスを指定した場合のredirect・通常マッチ処理。
- バフの`delay` / `lasting`による有効ラウンド管理。

## 3. 実装状況と残る不足

### 3.1 キャラクター固定属性

PR 1で`tag_ids`と`disabled_tag_ids`の正式なデータ契約を実装した。キャラクタービルダー、持ちキャラ保存、ルーム投入、戦闘専用プリセット、ルームプリセット、召喚は共通の正規化処理を使用する。出身・ボーナス国と輝化スキル由来タグの再生成、GM作成scenarioキャラクターの自由タグ、キャラクター詳細のタグ表示も実装済みである。

残る不足は、有効タグをconditionとPvE対象候補へ接続する処理であり、PR 2とPR 4で扱う。PR 1の実装済み仕様の正本は`C01_JSON_Definition_Master.md` §13と`E02_UI_Component_Specs.md` Part 7とする。

### 3.2 condition

現行`condition`は数値比較を主としており、`EQUALS`は最終的に整数変換を行う。キャラクターの`name`、`baseName`、`tag_ids`を文字列・配列として安定して参照する経路が不足している。

`CONTAINS`自体は存在するが、文字列部分一致と配列要素判定の正式仕様、ビルダー上の案内、回帰テストが不足している。

### 3.3 最大HPダメージ

対象の`maxHp`を参照し、倍率や軽減を無視して同値のダメージを与える汎用効果は存在しない。スキルごとの専用効果ではなく、再利用可能な`CUSTOM_EFFECT`として追加する必要がある。

### 3.4 死亡原因

現行の死亡時処理は死亡したキャラクターを起点に実行されるが、死亡原因となったスキル使用者と使用スキルを受け取らない。そのため、瓦礫側の死亡時パッシブから「ボスが壊したか」「PCが壊したか」を安全に判定できない。

また、HP更新が既に0のキャラクターへ再度行われた場合にも死亡処理へ入る余地がある。今回の要件では、`old_hp > 0`かつ`new_hp <= 0`の遷移だけを死亡として扱う。

### 3.5 PvE対象選択

現行の行動フローチャートは、各行動に対して単一の対象選択方法を保持する。次のような順序付き候補は表現できない。

1. 同陣営の「種別:瓦礫」を持つキャラクター。
2. 第一候補がいなければ、敵陣営の生存キャラクター。

また、複数行動スロットを持つ同一キャラクターについて、現在のスロットだけでなく同じ行動者の全スロットを自己対象候補から除外する必要がある。

## 4. 決定済み仕様

### 4.1 タグの保存形式

すべての固定属性は、キャラクターJSONの同じ`tag_ids`配列へ保存する。GMが一時的に無効化したタグは削除せず、`disabled_tag_ids`にも格納する。

```json
{
  "tag_ids": [
    "出身:ヨキューク・ツォー",
    "種族:シュラーカー",
    "種別:瓦礫"
  ],
  "disabled_tag_ids": [
    "種別:瓦礫"
  ]
}
```

保存時の共通正規化は次のとおり。

- 配列以外は空配列として扱う。
- 各要素を文字列化して前後空白を除去する。
- 空文字を除外する。
- 最初の出現順を保ったまま完全一致の重複を除去する。
- 大文字・小文字や全角・半角を暗黙変換しない。
- 既存データに`tag_ids`がなくても読み込める。

`disabled_tag_ids`の保存時は、`tag_ids`と同じ文字列正規化を行ったうえで次を適用する。

- `tag_ids`に存在するタグだけを保持する。
- 無効化はタグの削除ではなく、条件判定上の一時停止として扱う。
- タグを`tag_ids`から削除した場合、同じタグを`disabled_tag_ids`からも削除する。
- 既存データに`disabled_tag_ids`がない場合は空配列として扱う。
- 条件判定、スキル効果、PvE対象候補で参照する有効タグは`tag_ids - disabled_tag_ids`とする。

### 4.2 GM作成NPC・エネミー

- GMはキャラクタービルダーでタグ名を自由入力できる。
- 入力UIは複数タグを個別に確認・削除できる形式とする。
- `種族:シュラーカー`、`種別:瓦礫`、`機械`など、接頭辞の有無を問わず保存できる。
- 1タグは前後空白を除去した後、最大25文字とする。文字数はUnicodeコードポイント数で数える。
- タグの個数上限は現段階では設けない。
- 空文字と改行を含むタグは受理しない。
- 自由タグを許可するかは、キャラクターJSON内の自己申告ではなく、保存・ルーム投入を行うユーザーのGM権限で判定する。
- キャラクタービルダーの各タグにGM用の有効・無効切り替えを設ける。
- 無効化されたタグは`tag_ids`に残し、`disabled_tag_ids`へ追加する。
- GM以外から送信された`disabled_tag_ids`は信用しない。

### 4.3 プレイヤーキャラクター

- プレイヤー向け画面に自由タグ入力欄は出さない。
- プレイヤー向け画面にタグの有効・無効切り替えは出さない。
- 出身設定とボーナス国設定から出身国タグを自動付与する。
- 任意取得タグは輝化スキルとして定義する。
- 輝化スキルの取得コストは通過点を消費する。
- 輝化スキルを解除した場合、そのスキルのコスト分だけ使用済み通過点が減り、残り通過点が戻る。
- 自動付与タグと輝化スキル由来タグは同じ`tag_ids`へ格納する。
- プレイヤー保存データの`tag_ids`は、出身と選択済み輝化スキルからサーバー側で再計算する。クライアントから送られた任意タグを信用しない。
- プレイヤー保存データの`disabled_tag_ids`は空配列とし、クライアントから送られた無効化指定を信用しない。

タグによる能力効果はタグ自身へ直接持たせず、取得元の輝化スキルの既存`effect`として処理する。タグは条件判定・分類のためのマーカーとする。

独立したタグマスターシートは作らない。輝化スキルのスプレッドシートに任意の「付与タグ」列を追加し、ローダーがキャッシュ上の`granted_tag_ids`へ正規化する。タグ名以外の説明文、表示色、効果はタグデータとして保持しない。

```json
{
  "granted_tag_ids": ["特性:機械知識"],
  "effect": {
    "type": "buff",
    "stat_mods": {
      "知識": 1
    }
  }
}
```

### 4.4 キャラクター詳細画面のタグ表示

- キャラクター詳細画面では、キャラクター名の直下にタグをまとめて表示する。
- タグは入力順を保ち、横並びから必要に応じて折り返す。
- 有効タグは通常の文字色で表示する。
- `disabled_tag_ids`に含まれるタグは、赤文字かつ取り消し線で表示する。
- タグがないキャラクターではタグ表示領域自体を出さない。
- 詳細画面は表示専用とし、初期実装ではタグの追加・削除・有効状態変更を行わない。
- GMによる追加・削除・有効状態変更はキャラクタービルダーのタグ入力欄で行う。
- 将来、戦闘中にタグを切り替える要件が出た場合は、GM専用の編集導線を詳細画面へ別途追加する。

### 4.5 出身国タグ

- キャラクターの`params`にある「出身」のIDから正式表示名を解決する。
- 「ボーナス」で別の国が設定されている場合、その国も`出身:国名`として追加する。
- 出身国とボーナス国が異なる場合、キャラクターは`出身:`タグを2つ持つ。
- 出身国とボーナス国が同じ場合は、共通正規化により同じタグを1件だけ保持する。
- プレイヤーキャラクター保存時とルーム投入時に自動再計算する。
- 出身とボーナスの各IDを独立して解決し、有効な側だけを追加する。双方が未設定または不明なら出身国タグを追加しない。
- NPC・エネミーは出身なしを許可する。

例:

```json
{
  "tag_ids": [
    "出身:ヨキューク・ツォー",
    "出身:アーク・ジェムリア"
  ]
}
```

### 4.6 文字列condition

`condition`の演算子を値の型に応じて次のように扱う。

| 値の型 | `EQUALS` | `CONTAINS` |
|---|---|---|
| 数値 | 数値の完全一致 | false |
| 文字列 | 文字列の完全一致 | 部分一致 |
| 配列 | 配列全体の一致は今回使用しない | 要素の完全一致 |

キャラクターに対して正式対応するparamは、少なくとも次の3つとする。

- `name`: ランタイム表示名。
- `baseName`: 連番付与前の基礎名。
- `tag_ids`: 固定属性タグ配列。

`condition`から`tag_ids`を参照する場合、保存されている全タグではなく、`disabled_tag_ids`を除外した有効タグ配列を返す。無効タグは表示上残るが、完全一致・包含判定には使用しない。

瓦礫判定の正式形は名前ではなくタグを使う。

```json
{
  "source": "target",
  "param": "tag_ids",
  "operator": "CONTAINS",
  "value": "種別:瓦礫"
}
```

名前の部分一致も汎用機能として使用可能にする。

```json
{
  "source": "target",
  "param": "baseName",
  "operator": "CONTAINS",
  "value": "瓦礫"
}
```

### 4.7 `DEAL_TARGET_MAX_HP_DAMAGE`

新しい汎用`CUSTOM_EFFECT`キーを次のように定義する。

```json
{
  "timing": "HIT",
  "type": "CUSTOM_EFFECT",
  "target": "target",
  "value": "DEAL_TARGET_MAX_HP_DAMAGE"
}
```

挙動は次のとおり。

- 対象の`maxHp`を取得する。
- 対象へ`maxHp`と同値のダメージを与える。
- 与ダメージ倍率、被ダメージ倍率、ダメージ軽減を無視する。
- HPを直接0へ設定するのではなく、ダメージイベントとして記録する。
- ダメージ発生元としてスキル使用者と使用スキルを保持する。
- 対象不在、最大HP不明、最大HPが0以下の場合は安全に不発とする。
- 既にHP0の対象へ実行しても死亡時処理を再発させない。

### 4.8 死亡時コンテキスト

HPダメージから死亡時処理まで、少なくとも次の情報を引き継ぐ。

- 死亡原因となったキャラクター。
- 死亡原因となったスキルデータまたはスキルID。
- ダメージ種別。

死亡時効果の評価では、既存の死亡キャラクターを`actor`、死亡原因となったキャラクターを`target`として扱えるようにする。これにより、既存の`relation`条件と通常の`target`指定を再利用する。

- 瓦礫から見て破壊者が敵陣営なら、PCが破壊した扱い。
- 瓦礫から見て破壊者が同陣営なら、ボスが破壊した扱い。
- 発生元が不明な死亡では、発生元依存の効果は実行しない。

死亡時処理は`old_hp > 0`かつ`new_hp <= 0`の場合だけ1回実行する。蘇生後に再び正数から0へ遷移した場合は、新しい死亡として再実行してよい。

### 4.9 瓦礫破壊時バフ

瓦礫側の死亡時パッシブで破壊者を判定し、次を付与する。

| 破壊者 | 付与対象 | 効果 | delay | lasting |
|---|---|---|---:|---:|
| PC陣営 | PC陣営の生存者全員 | スキル最終威力+1 | 1 | 1 |
| ボス陣営 | 死亡原因となったキャラクター | スキル最終威力+1 | 1 | 2 |

したがって、PC側は次ラウンドのみ、ボス側は次ラウンドから2ラウンド有効となる。

### 4.10 PvEの順序付き対象候補

行動フローチャートの各行動に、複数の対象候補を優先順で設定できるようにする。

各候補は少なくとも次を持つ。

- 対象陣営: 同陣営 / 敵陣営 / 全員。
- 必須タグ: 0件以上。
- 選択方法: ランダム / 最速 / 最遅。

選択アルゴリズム:

1. 第一候補から順に、生存・配置済み・対象可能なキャラクターを抽出する。
2. `disabled_tag_ids`を除いた有効タグに対して、必須タグをすべて持つキャラクターだけを残す。
3. 行動者自身の全行動スロットを除外する。ただし明示的な自己対象設定は除く。
4. 候補が1体以上なら、その候補の選択方法で対象を決定して終了する。
5. 候補が0体なら次の候補へ進む。
6. すべて空なら対象なしとして行動を不発にする。別スキルへ差し替えない。

既存の単一対象設定は「候補が1件だけある定義」として読み込み、既存行動フローチャートとの後方互換性を保つ。この設定は行動フローチャート内部のデータであり、スキルJSONへ追加しない。

### 4.11 「鉄塊を捥ぎり喰う」

スキル本体の理想仕様:

```text
名称：鉄塊を捥ぎり喰う
分類：物理・近接・打撃
威力：9+1d1
対象範囲：any
使用ラウンド：3、6、9、12……
```

スキルJSONの中核:

```json
{
  "schema": "skill_json_rule_v2",
  "target_scope": "any",
  "effects": [
    {
      "timing": "HIT",
      "type": "CUSTOM_EFFECT",
      "target": "target",
      "value": "DEAL_TARGET_MAX_HP_DAMAGE",
      "condition": {
        "source": "target",
        "param": "tag_ids",
        "operator": "CONTAINS",
        "value": "種別:瓦礫"
      }
    }
  ]
}
```

行動フローチャートの対象候補:

1. 同陣営かつ`種別:瓦礫`を持つ対象からランダム。
2. 第一候補がいなければ、PC陣営からランダム。

期待挙動:

- 瓦礫が存在すれば瓦礫を狙う。
- 瓦礫が存在しなければ、同じスキルのままPCを狙う。
- PCが吸収行動より高速な行動でボスを指定した場合、通常どおりredirectされてマッチする。
- マッチ後の対象がPCなら瓦礫条件はfalseとなり、通常の`9+1d1`攻撃だけを解決する。
- 瓦礫へ命中した場合だけ`DEAL_TARGET_MAX_HP_DAMAGE`を追加実行する。
- 瓦礫の死亡時パッシブが破壊者を判定し、陣営別バフを次ラウンドから付与する。
- 3ラウンド周期は、行動フローチャートを3ステップで循環させ、3番目のステップにこのスキルを配置して表現する。

## 7. 実装詳細

### 7.1 タグ正規化の単一化

JSとPythonで別々の判断を増やさず、サーバー側の正規化結果を正とする。クライアントは入力補助と即時表示を行うが、保存時には必ずサーバーで再計算する。

概念的なPython API:

```python
def normalize_tag_ids(raw_values):
    """Return trimmed, non-empty, stable-order unique tag strings."""


def normalize_disabled_tag_ids(raw_values, tag_ids):
    """Return normalized disabled tags restricted to existing tag_ids."""


def derive_player_tag_ids(character_data, radiance_catalog):
    """Rebuild trusted tags from origin and selected radiance skills."""
```

プレイヤーキャラクターは、受信した`tag_ids`へ追記する方式ではなく、毎回再構築する。これにより輝化スキル解除時にタグが残留せず、JSON直接編集による任意タグ混入も防止できる。プレイヤーの`disabled_tag_ids`は常に空配列へ正規化する。

GM作成NPC・エネミーでは、`disabled_tag_ids`を`tag_ids`の部分集合として保存する。条件判定側が個別に差集合を計算しないよう、有効タグ取得処理も同じモジュールへ集約する。

### 7.2 通過点返還

通過点は現在選択中の輝化スキルのコスト合計から都度算出する。別の消費履歴をタグ用に新設しない。

```text
残り通過点 = params.通過点 - 現在選択中の輝化スキルコスト合計
```

輝化スキルを選択解除すると`SPassive`からIDが外れ、次回計算時に使用済み通過点と`granted_tag_ids`の両方が減る。クライアントの確認ダイアログで上限超過を許す現行挙動だけに依存せず、持ちキャラ作成・更新APIでも上限超過を拒否する。

### 7.3 conditionの型別比較

`EQUALS`で無条件に整数変換しない。両方が数値として扱える場合だけ数値比較し、それ以外は文字列比較を行う。

`CONTAINS`は次の順序で処理する。

1. 現在値が文字列なら、検査値を文字列化して部分一致。
2. 現在値がlist/tuple/setなら、要素の完全一致。
3. その他はfalse。

未知のparamや対象不在は従来どおりfalseとし、例外にしない。

### 7.4 最大HPダメージ

`DEAL_TARGET_MAX_HP_DAMAGE`は、既存の`CUSTOM_DAMAGE`適用経路を利用できる場合はそれを使う。ただし、倍率処理へ再投入されないことをテストで固定する。

ログには少なくとも効果名、対象名、ダメージ値を残す。

```text
《DEAL_TARGET_MAX_HP_DAMAGE》瓦礫に30ダメージ
```

### 7.5 死亡時効果のactor/target

死亡時パッシブでは次の対応にする。

```text
actor  = 死亡したキャラクター
target = 死亡原因となったキャラクター
target_skill_data = 死亡原因となったスキル
```

これにより、新しい死亡専用condition構文を増やさず、既存の`relation.target_is_ally` / `relation.target_is_enemy`と`target`指定を再利用できる。

### 7.6 PvE対象候補の後方互換

現行の行動フローチャート保存データにある単一の対象指定文字列は引き続き受理する。新形式は既存の各行動の対象設定を拡張して複数候補を持たせるが、内部フィールドをスキルJSONへ混入させない。

新UIでは各行動に対して次を表示する。

```text
対象候補 1  [同陣営] [必須タグ: 種別:瓦礫] [ランダム]
対象候補 2  [敵陣営] [必須タグ: なし]       [ランダム]
```

候補順はドラッグ操作に限定せず、上へ・下へ移動するボタンも用意する。

## 8. 互換性・移行

- `tag_ids`がない既存キャラクターは空配列として読み込む。
- `disabled_tag_ids`がない既存キャラクターは空配列として読み込む。
- プレイヤーの既存持ちキャラは、次回取得・保存・ルーム投入時に出身と輝化スキルからタグを再計算する。
- GM作成NPCの未知タグは削除せず、そのまま保持する。
- GM作成NPCの無効タグは表示上残し、conditionとPvE対象候補からは除外する。
- 既存の輝化スキルに`granted_tag_ids`がなくても効果は従来どおり動作する。
- 既存行動フローチャートの単一対象指定は自動的に候補1件として扱う。
- 死亡原因を渡さない既存呼び出しは引き続き動作し、発生元依存の死亡時効果だけが不発になる。
- `target_scope: same_team`の既存支援スキルとredirect禁止仕様は変更しない。「鉄塊を捥ぎり喰う」は`target_scope: any`を使う。

## 9. テスト計画

### 9.1 タグ

- 空、欠損、重複、空白入りタグの正規化。
- 出身ID・ボーナス国IDから出身国タグを自動付与。
- 出身国とボーナス国が異なる場合、`出身:`タグを2件付与。
- 出身国とボーナス国が同じ場合、同一タグを1件へ正規化。
- 出身不明ではタグを付与しない。
- 輝化スキルの追加でタグを付与。
- 輝化スキルの解除でタグと使用済み通過点を戻す。
- 通過点超過をサーバーが拒否。
- プレイヤーが送信した任意タグを保存しない。
- GMの自由タグを保存・復元。
- GMがタグを無効化すると`disabled_tag_ids`へ保存され、再度有効化すると削除される。
- `tag_ids`にない値を`disabled_tag_ids`へ送信しても保存されない。
- プレイヤーから送信された`disabled_tag_ids`を保存しない。
- タグ削除時に同じ値を`disabled_tag_ids`からも削除。
- GM自由タグは25文字を受理し、26文字以上を拒否。
- GM自由タグの個数だけを理由に拒否しない。
- ルーム投入、敵プリセット、召喚でタグを維持。

### 9.2 condition

- 文字列`EQUALS`の一致・不一致。
- 文字列`CONTAINS`の部分一致・不一致。
- `tag_ids CONTAINS`の要素一致。
- 無効タグは`tag_ids CONTAINS`に一致しない。
- `瓦礫`と`種別:瓦礫`を誤って同一視しない。
- 数値conditionの既存挙動に回帰がない。
- 対象不在・未知paramはfalse。

### 9.3 最大HPダメージ

- `maxHp=30`の対象へ30ダメージ。
- 現在HPが30未満でもダメージ値は30。
- 与ダメージ増減・被ダメージ軽減を無視。
- 最大HP不明・0以下では不発。
- 対象が瓦礫タグを持たない場合、スキルconditionにより実行されない。

### 9.4 死亡時処理

- HP正数から0で1回だけ発動。
- HP0へ追加ダメージを与えても再発しない。
- 蘇生後に再度0になれば再発する。
- PC破壊時はPC陣営全員へ`delay=1 / lasting=1`。
- ボス破壊時は破壊者へ`delay=1 / lasting=2`。
- 発生元不明では陣営別バフを付与しない。
- 通常攻撃、CUSTOM_DAMAGE、広域攻撃から発生元を保持する。

### 9.5 PvE・UI

- 第一候補に瓦礫がいれば瓦礫を選ぶ。
- 瓦礫タグが無効化されていれば第一候補に含めない。
- 瓦礫が複数ならその中からランダムに選ぶ。
- 瓦礫がいなければ第二候補のPCを選ぶ。
- ボス自身の別行動スロットを対象にしない。
- 3R、6R、9Rで対象スキルを選ぶ。
- 高速PCがボスを指定すると通常マッチになる。
- マッチ後にPCが対象なら最大HPダメージを発動しない。
- 敵インテント矢印が瓦礫またはPCへ正しく描画される。
- キャラクター詳細画面でタグが名前直下に表示される。
- 無効タグが赤文字・取り消し線で表示される。
- タグなしでは不要な空欄を表示しない。
- 長いタグと多数タグがデスクトップ・モバイル幅で折り返され、名前や後続項目と重ならない。
- 既存の行動フローチャートを編集・保存しても意味が変わらない。

```bash
python scripts/check_text_encoding.py
python scripts/check_mojibake_markers.py
pytest -q
npm run build
git diff --check
```

`npm run build`は`static/js/`を変更するPRで必須。キャラクタービルダーと戦闘画面は、デスクトップ・モバイル幅で実機確認する。
