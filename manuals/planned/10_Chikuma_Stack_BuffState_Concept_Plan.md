# 10 蓄魔（状態異常スタック型バフ）構想マニュアル

**作成日**: 2026-03-29  
**対象**: 新規スタック効果 `蓄魔`（状態異常として増減し、分類はバフ寄りで扱う）  
**位置づけ**: 実装前の大枠設計（仕様合意用）

---

## 1. 目的

`蓄魔` を「状態異常スタックの計算しやすさ」と「バフ運用の分かりやすさ」を両立した新カテゴリとして導入する。  
本構想では、以下を一貫して扱えることを目標にする。

1. スキル効果で `蓄魔` を獲得（加算）
2. スキル使用時・命中時などで `蓄魔` を消費
3. 消費量に応じて、バフ付与・FP回復・MP回復などへ変換

---

## 2. 基本コンセプト

## 2.1 立ち位置

- 実体: `states` 上の数値スタック（例: 出血と同じく `state_name` + 数値）
- 分類: UI/説明/効果参照では「バフ寄り」カテゴリとして扱う
- 方針: **保持は state、意味づけは buff-like**

この方針により、既存 `APPLY_STATE` 系を再利用しつつ、プレイヤー体験上は「育てて使うリソースバフ」として見せられる。

## 2.2 期待するプレイ体験

- 複数ターンで `蓄魔` を溜める
- ここぞのタイミングで消費して爆発力や継戦力に変換する
- 「今使うか、さらに溜めるか」の意思決定を生む

---

## 3. 仕様の骨格

## 3.1 データ表現（v1案）

- 保存先: `char["states"]["蓄魔"]`（整数、0以上）
- 初期値: 0
- 上限: `max_chikuma`（暫定 99、バランス調整で変更可）
- 0未満: 不可（消費時は下限0でクランプ）

## 3.2 分類メタ（buff-like）

`蓄魔` を単なるデバフ/状態異常として誤処理しないため、分類メタを持たせる。

- `status_class`: `stack_resource`
- `display_group`: `buff_like`
- `is_negative`: `false`

用途:

- UIでバフ欄寄せ表示
- 「弱体解除」など負の状態だけを対象にする処理から除外
- 将来の `status_stack_filter` 拡張時に明示的に参照

## 3.3 付与・増加

初期導入では既存 effect を優先再利用する。

- 推奨: `APPLY_STATE` / `APPLY_STATE_PER_N` で `state_name: "蓄魔"`
- 付与元: self / target はスキル設計に準拠

例:

```json
{
  "timing": "HIT",
  "type": "APPLY_STATE",
  "target": "self",
  "state_name": "蓄魔",
  "value": 2
}
```

## 3.4 消費・変換（新規効果の最小セット）

`蓄魔` の価値は「消費して変換できること」にあるため、v1では次の2種類を先行導入する。

1. `CONSUME_STATE`
- 指定 state を最大 N 消費する共通部品
- 実消費量を後続効果で参照可能にする

2. `CONVERT_CONSUMED_STATE`
- 直前に消費した量を、指定先へ変換
- 変換先: `buff`, `FP`, `MP`

例（3消費してFPを得るイメージ）:

```json
{
  "timing": "HIT",
  "type": "CONSUME_STATE",
  "target": "self",
  "state_name": "蓄魔",
  "consume_max": 3,
  "store_as": "consumed_chikuma"
},
{
  "timing": "HIT",
  "type": "CONVERT_CONSUMED_STATE",
  "target": "self",
  "source_key": "consumed_chikuma",
  "to": "FP",
  "rate": 1
}
```

補足: まずはこの2段構成で統一し、将来はショートハンド（`CONSUME_CHIKUMA_FOR_FP` など）を必要に応じて追加する。

---

## 4. ルール方針（ゲーム裁定）

## 4.1 消費ルール

- 消費要求 > 所持スタック の場合は「可能な分だけ消費」
- 実消費0なら変換効果は不発（ログには残す）
- 複数効果連鎖時は、記述順で消費・変換

## 4.2 解除・無効化との関係

- `蓄魔` は負状態解除（デバフ解除）の対象外
- 汎用 `states` 全消去系を使う場合は個別除外オプションを推奨
- 将来、`stack_resource` のみを対象とする吸収/奪取効果を拡張可能にする

## 4.3 表示ルール

- 戦闘UIでは「バフ寄り」欄に `蓄魔 xN` 表示
- ログ文言は「獲得」「消費」「変換」の3種を明示
- 説明文では「状態異常スタック型リソース」と定義

---

## 5. 実装フェーズ（大枠）

## Phase A: 仕様土台

- `蓄魔` の分類メタ（`stack_resource` / `buff_like`）を仕様書に追加
- effect JSON の新規2種（`CONSUME_STATE`, `CONVERT_CONSUMED_STATE`）を定義

## Phase B: ロジック実装

- `process_skill_effects` に消費・変換処理を追加
- 消費量の一時保持（`store_as`）を導入
- FP/MP加算、バフ付与への変換を実装

## Phase C: UI/ログ

- `蓄魔` の表示グループをバフ寄りへ
- 消費量・変換量が追える戦闘ログを追加

## Phase D: データ運用

- Buff Catalog / Glossary / スキル正本への入力ルールを整備
- シート更新 -> DB/キャッシュ反映手順を案内

---

## 6. テスト観点（大枠）

1. `APPLY_STATE` で `蓄魔` が加算される
2. 消費量が所持量を超えても負値にならない
3. 消費0時は変換が走らない
4. 消費量に応じて FP/MP が正しく増える
5. 消費量に応じて指定バフが正しく付与される
6. `蓄魔` がデバフ解除対象に誤って含まれない
7. UI で `蓄魔` がバフ寄り表示になる

---

## 7. リスクと先回り

- リスク: `states` と `buffs` の二重管理で扱いが曖昧になる  
  対応: 分類メタを必須化し、対象フィルタの既定値を文書化する

- リスク: 消費量参照（`store_as`）のスコープ漏れ  
  対応: 1スキル解決内のローカルコンテキストに限定する

- リスク: ログ/表示でプレイヤーに伝わらない  
  対応: 「現在値」「今回消費」「今回変換」を固定フォーマット化する

---

## 8. 受け入れ条件（構想完了基準）

1. `蓄魔` の立ち位置（state実体 + buff-like分類）が仕様として明文化されている
2. 獲得・消費・変換の最小ループが effect 定義で表現できる
3. FP/MP変換とバフ変換の両系統を同一仕組みで扱える
4. UI/ログ/データ運用まで含めて、実装着手可能な粒度で整理されている
