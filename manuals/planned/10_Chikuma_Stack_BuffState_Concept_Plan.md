# 10 蓄魔（状態異常スタック型バフ）構想マニュアル

**作成日**: 2026-03-29  
**最終更新日**: 2026-03-30  
**対象**: 新規スタック効果 `蓄魔`（状態異常として増減し、分類はバフ寄りで扱う）  
**位置づけ**: 実装前の大枠設計（仕様合意用）

---

## 1. 目的

`蓄魔` を「状態異常スタックの計算しやすさ」と「バフ運用の分かりやすさ」を両立した新カテゴリとして導入する。  
本構想では、以下を一貫して扱えることを目標にする。

1. スキル効果で `蓄魔` を獲得（加算）
2. スキル使用時・命中時などで `蓄魔` を消費
3. 消費量に応じて、バフ付与・FP回復・MP回復などへ変換

---

## 2. 基本コンセプト

## 2.1 立ち位置

- 実体: `states` 上の数値スタック（例: 出血と同じく `state_name` + 数値）
- 分類: UI/説明/効果参照では「バフ寄り」カテゴリとして扱う
- 方針: **保持は state、意味づけは buff-like**

この方針により、既存 `APPLY_STATE` 系を再利用しつつ、プレイヤー体験上は「育てて使うリソースバフ」として見せられる。

## 2.2 期待するプレイ体験

- 複数ターンで `蓄魔` を溜める
- ここぞのタイミングで消費して爆発力や継戦力に変換する
- 「今使うか、さらに溜めるか」の意思決定を生む

---

## 3. 仕様の骨格

## 3.1 データ表現（v1案）

- 保存先: `char["states"]["蓄魔"]`（整数、0以上）
- 初期値: 0
- 上限: `max_chikuma`（暫定 99、バランス調整で変更可）
- 0未満: 不可（消費時は下限0でクランプ）

## 3.2 分類メタ（buff-like）

`蓄魔` を単なるデバフ/状態異常として誤処理しないため、分類メタを持たせる。

- `status_class`: `stack_resource`
- `display_group`: `buff_like`
- `is_negative`: `false`

用途:

- UIでバフ欄寄せ表示
- 「弱体解除」など負の状態だけを対象にする処理から除外
- 将来の `status_stack_filter` 拡張時に明示的に参照

## 3.2.1 既存5状態異常の分類固定（回帰防止規約）

既存の状態異常5種類（`出血`・`破裂`・`亀裂`・`戦慄`・`荊棘`）は、分類を次で固定する。

| 状態異常 | status_class | display_group | is_negative |
| --- | --- | --- | --- |
| 出血 | `debuff` | `debuff` | `true` |
| 破裂 | `debuff` | `debuff` | `true` |
| 亀裂 | `debuff` | `debuff` | `true` |
| 戦慄 | `debuff` | `debuff` | `true` |
| 荊棘 | `debuff` | `debuff` | `true` |

補足:

- `蓄魔` だけが例外的に `stack_resource / buff_like / false` を取る
- デバフ解除・表示グループ判定・フィルタ判定は上表を正とし、実装時に暗黙推定しない

## 3.3 付与・増加

初期導入では既存 effect を優先再利用する。

- 推奨: `APPLY_STATE` / `APPLY_STATE_PER_N` で `state_name: "蓄魔"`
- 付与元: self / target はスキル設計に準拠

例:

```json
{
  "timing": "HIT",
  "type": "APPLY_STATE",
  "target": "self",
  "state_name": "蓄魔",
  "value": 2
}
```

## 3.4 消費・獲得量の個別定義（新規効果）

`蓄魔` の消費要求と獲得量は、**必ず別フィールドで数値定義**する。  
v1では、原子処理の単一 effect を導入する。

- `CONSUME_STATE_FOR_GAIN`
- `consume_required`: 必要な蓄魔量（消費要求）
- `gains`: 成功時に得る値（FP/MP/buff など）
- 判定: `current_stacks >= consume_required` のときのみ成功

例1（蓄魔3を消費してFP+1）:

```json
{
  "timing": "HIT",
  "type": "CONSUME_STATE_FOR_GAIN",
  "target": "self",
  "state_name": "蓄魔",
  "consume_required": 3,
  "gains": [
    { "type": "FP", "value": 1 }
  ]
}
```

例2（蓄魔4を消費してFP+2）:

```json
{
  "timing": "HIT",
  "type": "CONSUME_STATE_FOR_GAIN",
  "target": "self",
  "state_name": "蓄魔",
  "consume_required": 4,
  "gains": [
    { "type": "FP", "value": 2 }
  ]
}
```

補足: `gains` は複数指定可能（例: FP+1 と MP+1 を同時獲得、またはバフ付与）。

---

## 4. ルール方針（ゲーム裁定）

## 4.1 消費ルール

- `所持蓄魔 >= consume_required` のときだけ effect 成功
- `consume_required > 所持蓄魔` の場合、**消費も獲得も一切発生しない（不発）**
- 不足時に「一部だけ消費して一部だけ獲得」は行わない（部分成功なし）
- 複数効果連鎖時は、記述順で各 effect を独立判定

## 4.2 解除・無効化との関係

- `蓄魔` は負状態解除（デバフ解除）の対象外
- 汎用 `states` 全消去系を使う場合は個別除外オプションを推奨
- 将来、`stack_resource` のみを対象とする吸収/奪取効果を拡張可能にする

## 4.3 表示ルール

- 戦闘UIでは、出血などと同様に**キャラ駒右上へアイコン表示**し、`蓄魔 xN` を重ねて表示
- 表示グループ上は buff-like 扱いだが、見た目は既存状態異常アイコン群と同系統で統一
- ログ文言は「獲得」「消費」「変換」の3種を明示
- 説明文では「状態異常スタック型リソース」と定義

## 5. 実装フェーズ（大枠）

## Phase A: 仕様土台

- `蓄魔` の分類メタ（`stack_resource` / `buff_like`）を仕様書に追加
- 既存5状態異常（出血・破裂・亀裂・戦慄・荊棘）の分類を `debuff / debuff / true` で固定
- effect JSON の新規1種（`CONSUME_STATE_FOR_GAIN`）を定義
- `consume_required` と `gains` の必須項目・型・範囲を固定

## Phase B: ロジック実装

- `process_skill_effects` に `CONSUME_STATE_FOR_GAIN` を追加
- 先に所持量チェックを行い、不足時は不発ログのみ返す
- 成功時のみ「消費 -> gains適用」を同一トランザクションで実行

## Phase C: UI/ログ

- `蓄魔` アイコンを既存状態異常と同じくキャラ駒右上へ表示
- 消費量・変換量が追える戦闘ログを追加

## Phase D: データ運用

- Buff Catalog / Glossary / スキル正本への入力ルールを整備
- シート更新 -> DB/キャッシュ反映手順を案内

## 6. テスト観点（大枠）

1. `APPLY_STATE` で `蓄魔` が加算される
2. `consume_required=3, gains FP+1` が数値どおり処理される
3. `consume_required=4, gains FP+2` が数値どおり処理される
4. 所持不足時は「消費なし・獲得なし」で不発になる
5. 成功時のみ蓄魔が減り、FP/MP/バフ獲得が実行される
6. `蓄魔` がデバフ解除対象に誤って含まれない
7. UI で `蓄魔` がキャラ駒右上アイコンとして表示される
8. 既存5状態異常（出血・破裂・亀裂・戦慄・荊棘）が `debuff / debuff / true` の分類を維持する

---

## 7. リスクと先回り

- リスク: `states` と `buffs` の二重管理で扱いが曖昧になる  
  対応: 分類メタを必須化し、対象フィルタの既定値を文書化する

- リスク: 不足時に消費だけ走る/獲得だけ走る不整合  
  対応: `CONSUME_STATE_FOR_GAIN` を原子処理にし、成功時のみ更新する

- リスク: ログ/表示でプレイヤーに伝わらない  
  対応: 「現在値」「今回消費」「今回変換」を固定フォーマット化する

---

## 8. 受け入れ条件（構想完了基準）

1. `蓄魔` の立ち位置（state実体 + buff-like分類）が仕様として明文化されている
2. 消費要求値と獲得量を別々の数値として effect 定義できる
3. 消費要求超過時は、消費も獲得も発生しないルールが明文化されている
4. FP/MP変換とバフ変換の両系統を同一仕組みで扱える
5. UI/ログ/データ運用まで含めて、実装着手可能な粒度で整理されている
6. 既存5状態異常の分類が `debuff / debuff / true` で固定記述されている
