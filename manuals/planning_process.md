# 計画策定プロセス

このプロジェクトでは、実装前に「一問一答形式の議論」で方針を固め、その決定事項を計画書に即時記録する方法を採用している。

---

## なぜこの方式か

- 15〜20の論点を一度に議論すると発散し、決定が曖昧になる
- 論点に依存関係があるため、順序を間違えると手戻りが起きる
- 「議論した」と「決まった」を分離しないと、後から見た人が判断できない

---

## 議論の進め方

### 1. 論点の整理（実装支援エージェントが担当）

実装前に論点全体を列挙し、**依存関係に基づいて順序を決める**。

優先順位の原則:
```
目的と範囲 → デザイン原則 → デザイントークン → コンポーネント規約 → 移行戦略
```

上位が決まらないと下位の評価軸が定まらないため、この順を守る。

### 2. 1ターン1論点

実装支援エージェントが以下の形式で提示する:

```
論点タイトル

選択肢:
- (1) ...  ← 内容 + trade-off
- (2) ...
- (3) ...

推奨: (X) 理由: ...
```

ユーザーは番号または短い補足で返答する。複数論点を1ターンで出す場合は、
**互いに依存しない論点のみ**まとめる（最大3問）。

### 3. 決定事項の即時記録

合意した瞬間に計画書の「決定事項ログ」テーブルに追記する。

```markdown
| 日付 | 論点 | 決定 | 根拠 |
|---|---|---|---|
| YYYY-MM-DD | 論点名 | **決定内容** | 1行で根拠 |
```

**記録しない情報:** 議論の途中経過、却下した選択肢の詳細。
それらはチャット履歴に残れば十分であり、計画書に書くと読みにくくなる。

### 4. 未決事項の管理

その場で決めない論点は「未決定事項」セクションに移す。
「後で決める」と明示しておくことで、実装者が勝手に判断することを防ぐ。

決定事項ログを更新した場合は、関連する未決定事項を同じタイミングで見直す。
- 決まった項目は未決定事項から削除する
- 一部だけ決まった項目は、残っている論点だけが分かる形に書き換える
- 実装済みかどうかも必要に応じて計画書へ反映する

---

## 計画書の構造ルール

計画書（`manuals/planned/XX_*.md`）には以下のセクションを設ける:

| セクション | 内容 |
|---|---|
| 目的 | 何のための計画か（1〜3行） |
| 対象範囲 | 触る画面・ファイル / 触らないもの |
| 実装段階 | Phase 分割と各Phaseの作業内容 |
| 推奨PR分割 | PR単位の作業範囲 |
| **実装詳細** | Codex等が迷わず着手できる具体的な作業内容（コード例含む） |
| **決定事項ログ** | 議論で決まったことの一覧（日付・根拠付き） |
| 未決定事項 | まだ決まっていないことの一覧 |
| 受け入れ条件 | 完了の定義 |

「実装詳細」と「決定事項ログ」は議論フェーズの後に追記する。
議論前の計画書には存在しなくてよい。

---

## Codex（または別エージェント）への引き継ぎ

計画書1本で引き継げる状態にするために、以下を確認する:

- [ ] 「実装詳細」セクションに対象ファイル名と具体的な変更内容が書かれているか
- [ ] コード例（CSS・JS等）が実際に動く形で記載されているか
- [ ] 「触ってはいけないもの」が明示されているか（機能変更禁止など）
- [ ] 決定事項ログが最新の状態か
- [ ] 未決定事項が残っている場合、その影響範囲が明示されているか

---

## この方式を使う場面

- 新機能の設計方針を固める前
- UI刷新・リファクタリングの方針決定
- 複数の実装選択肢がある場合

**使わない場面:**
- バグ修正（すでに「何をすべきか」が明確）
- 1ファイル以内で完結する小さな変更
