## はじめに

このマニュアルでは、GM（ゲームマスター）およびシナリオ・データ作成者向けに、ツールの高度な機能やデータ定義仕様について解説します。
プレイヤー向けの内容に加えて、ルーム管理、スキル作成、バフ/アイテムの定義方法を含みます。

## 1. GM権限とルーム管理

権限はルーム単位の **ロール（owner / gm / player）** と、アプリ全体の **アプリ管理者権限** に分離されています。

### 1.1 ルームロールとメンバーシップ

ルーム内権限はメンバーシップ（DB上のロール）で管理されます。ルーム作成者が自動的に owner になります。

| ロール | 主な権限 |
|---|---|
| owner | ルーム削除、owner移譲、GM付与/解除、参加コード管理、ルーム設定（公開設定・説明・募集）編集 |
| gm | GM専用戦闘操作（バフ付与・ラウンド進行等）、募集状態編集 |
| player | 参加者向け状態閲覧、自キャラ操作、チャット |

ルームへGMとして入室するには、入室時に **4桁GM PIN** を入力します。GM PINを知っていれば player でも gm membership を取得できます。8桁マスターキーでのGM入室も有効です。

- ロビーへ戻るとルームGM状態は解除されます。
- クライアント送信の `attribute=GM` はサーバー側の membership 確認なしでは信用されません。
- アプリ管理者がルームへ入室しても、**自動的にGMにはなりません**（GM PINまたはマスターキーが必要）。

### 1.2 アプリ管理者権限

アプリ管理者権限は、ユーザー管理とパスワードリセット操作に使う権限です。ルームGMロールとは完全に分離されています。

- ユーザー管理画面の一覧/詳細は **アプリ管理者のみ** 閲覧できます。
- ユーザー削除・所有権譲渡はアプリ管理者のみ実行できます。
- **パスワードリセット用ワンタイムコード**の発行はアプリ管理者のみ実行できます。コードは発行時の一度だけ表示され、10文字・15分有効・5回失敗で失効します。ユーザーはコードを使って新しいパスワードを設定できます。
- アプリ管理者権限の付与/解除はマスターキー入力を必要とします。

### 1.3 マスターキー

マスターキーは8桁数字で、環境変数 `GM_MASTER_KEY` に設定します。

- 未設定の場合、マスターキー機能は無効です。
- **初回セットアップ**: 最初のアカウントを新規登録後、`/api/admin/set_user_management_admin` にマスターキーを使って自分にアプリ管理者権限を付与します。以降はアプリ管理者がユーザー管理を担当します。
- マスターキーはルームGM入室（GM PIN代替）にも使えます。
- `SECRET_KEY` とは別の値です。

GM権限でルームに入ると、プレイヤー画面にはない管理用コントロールが表示されます。

### 1.4 ルームメンバーシップ管理

ownerはロビーのルームカードにある「⚙️設定」ボタンからルームを管理できます。

**ルーム公開設定 (`lobby_visibility`)**

| 設定 | 効果 |
|---|---|
| `hidden` | 未参加者のロビー一覧に表示しない |
| `listed` | 安全なカード（ルーム名・説明・募集状態）のみ表示し、参加コード入力を許可 |
| `closed` | カードは表示するが新規参加を拒否 |

**参加コード管理**（owner専用）

- 「参加コード設定」でコードを手動指定するか、省略すると自動生成します（4〜32文字）。
- 「再発行」で既存コードを失効し新しいコードを発行します。
- 「失効」でコードを削除し、既存メンバー以外は参加できなくなります。
- コード実値は設定/再発行時の一度だけ表示されます（owner以外には非表示）。

**メンバー操作**（owner専用）

| 操作 | 内容 |
|---|---|
| GM付与 | player → gm に昇格 |
| GM解除 | gm → player に降格 |
| owner移譲 | 自分の owner を別メンバーへ移す（自分は gm になる） |
| メンバー除名 | player または gm をルームから除名 |

### 1.5 アカウント管理

ロビーのユーザー設定から変更できます。

- **表示名変更**: ログインIDとは独立した表示名をいつでも変更できます。
- **パスワード変更**: 現在のパスワードを確認してから新しいパスワードを設定します（10〜128文字）。

### ルーム状態の管理 (Room Controls)

#### ラウンド進行の制御

* **次R開始**: タイムライン上部のボタンで、新しいラウンドを開始し、全員のSpeed判定を自動で振ります。
* **ラウンド強制終了**: 現在のラウンドを強制的に終了させ、終了時効果（バフ減少など）を適用します。

#### ルームリセット (Room Reset)

セッションの終了時や、新しい戦闘を開始する際にルームの状態を初期化できます。
**設定タブ** または **管理メニュー** から「ルームリセット」を選択します。

**リセット可能な項目:**

1. **ラウンドリセット**: ラウンド数を1に戻し、現在の手番情報をクリアします。
2. **ステータス全回復**: 全キャラクターの<span style="color:#f1c40f; font-weight:bold;">HP</span>・<span style="color:#f1c40f; font-weight:bold;">MP</span>を最大値まで回復します。
3. **状態異常解除**: 付与されている全てのバフ・デバフを除去します。
4. **ログ消去**: 画面上のチャットログ・システムログをクリアします。
5. **配置リセット**: 全キャラクターを未配置状態(座標 -1, -1)に戻します。

### キャラクター管理・操作

#### 配置と移動 (探索モード/戦闘モード)

* **戦闘モード**: グリッド(90px)単位でスナップします。配置ボタンを押すと(0,0)または空いている場所に配置されます。
* **探索モード**: ピクセル単位で自由に配置可能です。**「配置」ボタンを押すと、現在表示している画面の中央に出現します。**（迷子防止仕様）

#### その他の操作

* **NPC追加**: GM用プリセットから敵キャラクターなどを素早く読み込めます。
* **代理操作**: GMは、全てのキャラクター（PLキャラ含む）のスキル選択・宣言・移動を代理で行うことができます。
* **強制移動**: マップ上の駒をドラッグ＆ドロップで強制的に移動できます。
* **ステータス編集**: キャラクター詳細画面から、非公開ステータス（hidden_skillsなど）の閲覧・編集が可能です。
* **行動チャート編集**: 敵キャラの設定から `behavior_profile`（行動ループ）をJSON編集できます。
* **プリセットJSON搬出入**: プリセット管理モーダルで Export / Import が可能です（ルーム間移送向け）。

## 2. データ定義の概要

本ツールはJSONベースのデータ定義により拡張可能です。
以下のカテゴリに関する詳細仕様は、別冊の **`C01_JSON_Definition_Master.md` (統合JSON定義マニュアル)** を参照してください。

* **スキル定義**: 特記処理 (Special Effects) の記述仕様
* **バフ定義**: `buff_id` 基準の静的/テンプレート定義、プラグイン定義
* **アイテム定義**: `items_cache.json` の構造
* **輝化・出身定義**: パッシブスキルおよび出身による自動効果
* **用語図鑑定義**: 図鑑シート列と埋め込み記法（`manuals/implemented/C01_JSON_Definition_Master.md`）
* **外部画像**: Cloudinary連携の設定方法。アップロード可能形式、サイズ上限、Render環境変数、確認手順は `manuals/implemented/F01_Operations_Manual.md` の「Part 1-C: 画像アップロード運用」を参照してください。

補助チュートリアル:
* **利用者向け用語図鑑ガイド**: `manuals/implemented/A04_Glossary_Tutorial.md`

### 2.1 JSON定義ビルダー（CharaCreator）追補: 震盪テンプレ

- `CharaCreator/json_definition_builder.html` の Effects タブに `震盪スキル雛形を生成` ボタンを追加。
- 押下時に以下を一括入力:
  - `cost`: `FP:2,MP:3`
  - `effects[0]`: `WIN` + `APPLY_BUFF(target)` + `buff_id="Bu-29"` + `lasting=3` + `data.count=2`
  - `effects[1]`: `HIT` + `APPLY_STATE(target)` + `state_name="破裂"` + `value=3`
- 生成JSONはそのまま `特記処理` 用の rule_data として利用可能。

## 追補（2026-02）: GM向け解決フェーズ運用

- 解決フェーズ表示はGM主導で全参加者に同期されます。
- 各処理は手動で「次のマッチへ」進行できます。
- 自動進行チェックON時は、各処理表示完了2秒後に次へ進みます。
- 解決表示中は事前ネタバレ防止のため、状態変化ログは段階表示後に反映されます。
- 全件表示完了後にまとめログをチャットへ送信し、`round_end` へ遷移します。
- 広域-合算は「防御威力合計」を基準に判定し、差分はダイス由来ダメージとして扱います。
- `battle_resolve_confirm` / `battle_resolve_flow_advance_request` は GM のみ実行できます。
- intent操作は対象スロットの操作権限チェックを通過した場合のみ反映されます（GMは全スロット代理操作可）。
- 無効な `slot_id` や権限不足は `battle_error`（`unknown slot_id` / `permission denied`）として返されます。

## 追補（2026-02-26）: 敵行動チャートとプリセット運用

### 1) 敵行動チャート（Behavior Profile）設定手順
1. 敵キャラクターの設定画面を開く。
2. `behavior_profile` を編集し、`enabled=true` と `initial_loop_id` を設定する。
3. `loops` に `steps.actions`（スキルID配列）と `transitions`（条件遷移）を設定する。
4. 各stepで「スキル使用後にループ遷移」を有効化すると、条件なしで遷移先ループを指定できる。
5. 遷移先はプルダウンで選択し、保存後の次ラウンドから反映される。
6. 保存後、PvEの自動intentでチャートが優先されることを確認する。

### 2) プリセットJSON搬出入
1. プリセット管理を開く。
2. ExportでJSONを出力し、別ルームでImportする。
3. Import時に検証エラーが出たJSONは保存されないため、構造を修正して再投入する。

### 3) 権限・安全性
- プリセット保存/読込/削除/搬出入はサーバー側でも GM 権限が必須です。
- v1旧形式プリセットは読込時に正規化され、v2形式として扱われます。

### 2.2 JSON定義ビルダー追補: 凝魔・蓄力 消費effect

- Effects Type に `CONSUME_BUFF_COUNT_FOR_GAIN` / `CONSUME_BUFF_COUNT_FOR_POWER` を追加。
- クイックテンプレに「凝魔3消費でMP+5」「凝魔最大5消費で最終威力上昇」を追加。
- `condition.param` ガイドに `蓄力_count` / `凝魔_count` / `buff_count:凝魔` を追記。
- 追加キー（`consume_required`, `gains`, `consume_max`, `value_per_stack`, `apply_to`, `min_consume`）は `extra_json` で入力する。

## 追補（2026-07-12）: プレイヤーの持ちキャラへの成果反映

プレイヤーがアカウントに保存した「持ちキャラ」（詳細: A01追補、F01 Part 9）をルームへ投入している場合、GMはそのキャラの詳細（⚙️設定メニュー）から「成果を反映」操作を、**キャラ本人の持ち主に代わって**実行できます。

- 経験値と（ホロウ以外の通常ルームでは）アイテムを、プレイヤーの持ちキャラへ書き戻せます。
- 1キャラ・1セッションにつき1回のみ反映されます（既に反映済みの場合はボタンの代わりに「反映済みです」と表示される）。
- 投入されたキャラでも、JSON貼り付けなど従来どおりの方法で追加されたキャラには「成果を反映」セクション自体が表示されません（持ちキャラ由来のキャラのみが対象）。

## 追補（2026-07-20）: NPC・エネミーのキャラクタータグ

キャラ作成ツールで`characterType`をscenarioにすると、GM用の自由タグ入力欄が表示されます。

1. 「キャラクタータグ」へ1件25文字以内でタグ名を入力し、「追加」を押します。
2. 一時的に無効状態として保存する場合は、そのタグの「有効」チェックを外します。
3. 不要なタグは削除ボタンで取り除きます。
4. JSONを保存し、持ちキャラ、プリセット、またはルーム投入に使用します。

タグの個数上限はありません。改行を含むタグと26文字以上のタグはサーバー側でも拒否されます。無効化したタグは削除されず、キャラクター詳細では赤文字・取り消し線で表示されます。自由タグの保存とルーム投入にはGM権限が必要です。conditionやPvE対象選択で無効タグを除外する処理は、計画37のPR 2・PR 4で実装予定です。
